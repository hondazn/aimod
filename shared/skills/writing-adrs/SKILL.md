# ADR 作成スキル

アーキテクチャ決定記録（ADR）を作成・改善する。ADRの本質は「未来の自分やチームメイトが、なぜその決定をしたかを理解できるようにする」ことにある。単なるフォーマットへの記入ではなく、意思決定の思考プロセスを残すことが目的。

## モード選択

入力に応じて以下のモードを判断:

| モード | トリガー | 出力 |
|--------|----------|------|
| **作成** | 「書いて」「作って」「記録して」+ 決定内容 | 完成したADRファイル |
| **レビュー** | 既存ADRが提示 + 「レビュー」「改善」 | 改善提案 + 修正版 |
| **テンプレート** | 「テンプレート」「ひな形」 | 空のテンプレート |

---

## 作成モード

### Step 1: プロジェクトの慣習を把握する

まず既存のADRがあるか確認する。プロジェクトには独自の書き方やルールがあることが多く、それに合わせることでチーム内の一貫性を保てる。

1. `docs/adr/`、`docs/decisions/`、`adr/`、`decisions/` を探す
2. 既存のADRがあれば、そのフォーマット・文体・構成を踏襲する
3. なければ `docs/adr/` への配置を提案し、ADR-0001から開始

### Step 2: 情報収集

以下を確認する。すべてが揃っていなくても構わない — ユーザーが持っている情報から始めて、足りない部分だけ質問する。一度に大量の質問を投げるのではなく、最も重要な不足情報に絞って聞く。

- **決定事項**: 何を決めたか/決めるか
- **背景**: なぜこの決定が必要になったか（ビジネス要件、技術的制約、チーム事情など）
- **検討した選択肢**: どんな代替案があったか（最低2つ）
- **選択理由**: なぜその選択肢を選んだか
- **影響**: この決定により何が変わるか（良い面も悪い面も）

### Step 3: ADR番号とファイル名の決定

既存ADRの最新番号 + 1 を使う。

- ファイル命名: `NNNN-kebab-case-title.md`（例: `0001-use-postgresql-for-database.md`）
- 番号は4桁ゼロ埋め
- 動詞で始める（use, adopt, implement, migrate など）

### Step 4: 執筆

下記のテンプレート集から、決定の重要度に応じて適切なものを選ぶ。

テンプレート選択の目安:
- **標準**: 重要な技術決定（DB選定、アーキテクチャ変更など）
- **シンプル**: 中程度の決定を素早く記録したい場合
- **RFC風**: チームでの正式な承認プロセスがある場合
- **軽量**: 日常的な小さな決定のログ

迷ったら**シンプルテンプレート**から始めて、必要に応じて詳細を追加するのが実用的。

執筆時に意識すること:

- **背景は第三者が読んで理解できるように書く** — 決定に関わった人だけが分かる前提知識に依存しない。6ヶ月後に入社した新メンバーがこのADRを読むことを想像する
- **選択肢は公平に評価する** — 採用した選択肢のメリットだけ、不採用の選択肢のデメリットだけを書くと、結論ありきに見えて信頼性が下がる。各選択肢のメリット・デメリットの両方を記載する
- **決定理由は背景・課題と論理的に繋げる** — 「良さそうだから」「流行っているから」ではなく、具体的な要件や制約にどう応えるかを書く
- **ネガティブな影響も正直に書く** — すべての技術決定にはトレードオフがある。ネガティブな影響を認識した上で軽減策を示すことで、決定の信頼性が高まる

### Step 5: ファイル作成

ADRを実際にファイルとして作成する:

1. 適切なディレクトリにMarkdownファイルを作成
2. 作成後、ユーザーに内容を確認してもらう
3. 追加の選択肢や背景情報があれば反映する

### 出力形式（作成）

```
[生成したADR]

---

### 作成メモ
- ファイル: [パス]
- ADR番号: ADR-NNNN
- ステータス: [提案/承認]
- 検討した選択肢: [N個]
```

## レビューモード

既存のADRを評価し、改善提案を行う。レビューの目的は「このADRを読んだ人が、決定の背景と理由を十分に理解できるか」を確認すること。

### 評価観点

| 観点 | 確認内容 | なぜ重要か |
|------|----------|-----------|
| **背景の明確さ** | 第三者が読んで課題を理解できるか | 背景が不明確だと、なぜこの決定が必要だったか分からない |
| **選択肢の公平性** | 各選択肢を客観的に評価しているか | 偏った記述は結論ありきに見え、信頼性が損なわれる |
| **理由の論理性** | 決定理由が背景・選択肢と整合しているか | 論理の飛躍があると、後から見直したときに納得感がない |
| **影響の具体性** | 影響が具体的に記述されているか | 曖昧な影響記述は実行可能なアクションに繋がらない |
| **将来の参照価値** | 1年後に読んで「なぜ」が分かるか | ADRの存在意義そのもの |

### 品質スコア

| スコア | 状態 |
|--------|------|
| 5 | 第三者が読んで完全に理解・納得できる。選択肢の比較が公平で、影響も具体的 |
| 4 | 概ね良好だが、軽微な補足があるとより良い |
| 3 | 一部の理由・影響が不明確で、追加の説明が欲しい |
| 2 | 選択肢の評価が偏っている、または重要な情報が欠落している |
| 1 | 背景・決定理由が欠落しており、ADRとしての役割を果たしていない |

### 出力形式（レビュー）

```markdown
## レビュー結果

### 品質スコア: X / 5

### 良い点
- [評価できる点]

### 改善が必要な点
1. **[問題の概要]** — [該当セクション]
   - 現状: [何が問題か]
   - 改善案: [具体的にどうすればよいか]

---

### 修正版

[改善を反映したADR全文]
```

## テンプレート集

用途に応じたテンプレートを選択する。ユーザーが迷っている場合は、決定の重要度とチームの承認プロセスの有無に基づいて推奨する。

### 標準テンプレート

重要な技術決定向け（目安: 30-60分）。選択肢の比較を丁寧に行い、決定の根拠を詳しく残す。

```markdown
# ADR-NNNN: [決定のタイトル]

## ステータス

提案 / 承認 / 廃止 / 置換（ADR-XXXXにより）

## 日付

YYYY-MM-DD

## 決定者

- [名前1]
- [名前2]

---

## コンテキスト

### 背景

[現在の状況と、なぜ決定が必要かを説明。
プロジェクト固有の文脈（ビジネス要件、技術的背景）を含める。
第三者が読んで状況を理解できるレベルの具体性を意識する。]

### 課題

- [解決すべき具体的な問題1]
- [解決すべき具体的な問題2]

### 制約

- [考慮すべき制約1（予算、チームスキル、期限など）]
- [考慮すべき制約2]

---

## 検討した選択肢

### 選択肢1: [名前]

**概要**: [選択肢の説明]

**メリット**:
- [メリット1]
- [メリット2]

**デメリット**:
- [デメリット1]
- [デメリット2]

### 選択肢2: [名前]

**概要**: [選択肢の説明]

**メリット**:
- [メリット1]
- [メリット2]

**デメリット**:
- [デメリット1]
- [デメリット2]

### 選択肢3: [名前]（任意）

[同様の形式]

---

## 決定

[選択肢X]を採用する。

### 理由

1. [理由1: 背景・課題との関連を明示]
2. [理由2: 他の選択肢との比較で優位な点]
3. [理由3: 制約との適合]

---

## 結果と影響

### ポジティブな影響

- [良い影響1]
- [良い影響2]

### ネガティブな影響

- [悪い影響1]（軽減策: [対策]）
- [悪い影響2]（許容範囲: [理由]）

### 必要なアクション

- [ ] [決定後に必要なアクション1]
- [ ] [決定後に必要なアクション2]

---

## 関連

- [ADR-XXXX: 関連する決定](./adr-xxxx.md)
- [設計ドキュメント](./design.md)
```

### シンプルテンプレート

迅速な記録向け（目安: 10-20分）。中程度の決定を短時間で文書化したい場合に使う。

```markdown
# ADR-NNNN: [決定のタイトル]

## ステータス

[提案 / 承認]

## 日付

YYYY-MM-DD

## コンテキスト

[なぜこの決定が必要か。プロジェクトの状況と課題を簡潔に記述。]

## 検討した選択肢

- **[選択肢A]**: [概要]（メリット: [要点] / デメリット: [要点]）
- **[選択肢B]**: [概要]（メリット: [要点] / デメリット: [要点]）

## 決定

[何を決定したか]

## 理由

- [理由1]
- [理由2]

## 影響

- [影響1]
- [影響2]
```

### RFC風テンプレート

チーム承認プロセスがある場合や大規模な技術変更向け（目安: 1-2時間）。トレードオフの比較表やリスク評価を含む。

```markdown

| 項目 | 内容 |

| **ステータス** | Draft / Review / Approved / Rejected |
| **作成者** | [名前] |
| **作成日** | YYYY-MM-DD |
| **レビュアー** | [名前1], [名前2] |
| **承認日** | YYYY-MM-DD |

## 代替案

### 代替案1: [名前]

[説明と、採用しない理由]

### 代替案2: [名前]

[説明と、採用しない理由]

## トレードオフ
```

### 軽量テンプレート

日常的な技術選択の記録向け（目安: 5分）。小さな決定でも記録として残しておくことで、後から「なぜこうしたか」を辿れるようにする。

```markdown
# [決定のタイトル]

**日付**: YYYY-MM-DD
**ステータス**: 承認

## 何を決めたか

[1文で決定内容]

## なぜ

[1-3文で理由。プロジェクト固有の文脈を含める。]

## 他の選択肢

- [選択肢A]: [採用しない理由]
- [選択肢B]: [採用しない理由]

## 影響

[1-2文で影響]
```

---

## ステータスの使い方

| ステータス | 意味 | いつ使うか |
|-----------|------|-----------|
| **提案** | 検討中 | ドラフト作成時。レビュー前 |
| **承認** | 決定済み | チームレビュー通過後 |
| **廃止** | 無効化 | 状況の変化により決定が不適切になった時 |
| **置換** | 別ADRで更新 | 新しいADRで決定を上書きした時。`置換（ADR-XXXXにより）`の形式で記載 |

## 品質の高いADRとは

ADRの品質は「6ヶ月後のチームメイトが、このドキュメントだけを読んで意思決定の背景と理由を理解できるか」で測れる。

**良いADRの特徴:**
- 背景が具体的で、プロジェクト固有の文脈（要件、制約、チーム構成）が含まれている
- 最低2つの選択肢が、それぞれメリット・デメリットの両面から公平に評価されている
- 決定理由が背景・課題と論理的に繋がっている
- ネガティブな影響が正直に書かれ、軽減策が示されている

**よくある問題とその対処:**

| よくある問題 | なぜ問題か | 対処 |
|-------------|-----------|------|
| 「〜だから採用」だけで比較なし | 他の選択肢を検討したか不明で、説得力がない | 最低2つの選択肢を比較する |
| メリットだけ/デメリットだけ | 結論ありきに見え、客観性が疑われる | 各選択肢の両面を書く |
| 「良さそう」「モダンだから」 | プロジェクト固有の理由がなく、再現性がない | 具体的な要件・制約との対応を書く |
| 影響を書かない | 決定のトレードオフが不明で、後の判断材料にならない | ポジティブ・ネガティブ両方の影響を書く |
| 内部用語のみ | 新メンバーやチーム外の人が理解できない | 初出時に簡潔に説明を添える |
