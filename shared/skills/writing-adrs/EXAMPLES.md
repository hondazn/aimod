# ADR 例集

良い例と悪い例を対比して示す。悪い例には改善版も併記しているので、何をどう直せばよいかが分かる。

---

## 良い例1: データベース選定（標準テンプレート）

背景・制約が具体的で、選択肢の比較が公平。ネガティブな影響に軽減策を添えている好例。

```markdown
# ADR-0001: PostgreSQLをメインデータベースとして採用

## ステータス

承認

## 日付

2024-01-15

## コンテキスト

### 背景

新規プロジェクトのデータベースを選定する必要がある。ユーザー情報、
取引履歴、商品データを管理し、将来的に分析基盤との連携も想定している。

### 課題

- トランザクションの整合性が重要（金融系データを扱う）
- 複雑なクエリが必要（レポート生成）
- 将来的なスケーラビリティ

### 制約

- チームにRDBMS経験者が多い
- 予算制約あり（マネージドサービスのコストを考慮）
- 初期データ量は小規模だが、年間2倍の成長を想定

---

## 検討した選択肢

### 選択肢1: PostgreSQL

**概要**: OSSのリレーショナルデータベース

**メリット**:
- ACID準拠で強い整合性
- 複雑なクエリに強い（Window関数、CTE等）
- JSONBでドキュメント的な使い方も可能
- マネージドサービス（RDS、Cloud SQL）が充実

**デメリット**:
- 水平スケーリングは自前で対応が必要
- NoSQLほどの書き込みスループットは出ない

### 選択肢2: MongoDB

**概要**: ドキュメント指向NoSQLデータベース

**メリット**:
- スキーマレスで柔軟
- 水平スケーリングが容易
- 書き込みスループットが高い

**デメリット**:
- トランザクションのサポートが限定的
- 複雑な結合クエリが苦手
- チームに経験者が少ない

### 選択肢3: MySQL

**概要**: OSSのリレーショナルデータベース

**メリット**:
- シンプルで学習コストが低い
- 実績が豊富
- マネージドサービスが充実

**デメリット**:
- PostgreSQLより複雑なクエリが苦手
- JSON機能がPostgreSQLより弱い

---

## 決定

PostgreSQLを採用する。

### 理由

1. **整合性要件**: 金融系データを扱うため、ACID準拠が必須。MongoDBは除外。
2. **クエリ複雑性**: レポート生成に複雑なクエリが必要。PostgreSQLのWindow関数、
   CTEが有効。MySQLより優位。
3. **柔軟性**: JSONBにより、スキーマ変更が頻繁な部分も対応可能。
4. **チームスキル**: RDB経験者が多く、学習コストが低い。

---

## 結果と影響

### ポジティブな影響

- 強い整合性により、金融データの信頼性を確保
- 複雑なレポートクエリを効率的に実行可能

### ネガティブな影響

- 将来的に書き込みスループットがボトルネックになる可能性
  （軽減策: リードレプリカ、キャッシュ層の追加）
- 水平スケーリングが困難
  （許容範囲: 当面は垂直スケーリングで対応可能）

### 必要なアクション

- [ ] AWS RDS PostgreSQL環境の構築
- [ ] バックアップ・リストア手順の策定
- [ ] 監視・アラートの設定
```

**この例が良い理由:**
- 課題と制約が具体的（「金融系データ」「年間2倍の成長」など固有の文脈がある）
- 3つの選択肢すべてにメリット・デメリットが記載されている
- 決定理由が課題・制約と論理的に対応している
- ネガティブな影響に軽減策が添えられている
- 必要なアクションが具体的

## 良い例2: フロントエンドフレームワーク選定（標準テンプレート）

```markdown
# ADR-0002: Reactをフロントエンドフレームワークとして採用

## ステータス

承認

## 日付

2024-01-20

## コンテキスト

管理画面のフロントエンドフレームワークを選定する。複雑なフォーム、
リアルタイム更新、ダッシュボードが主な機能。

---

## 検討した選択肢

### 選択肢1: React + TypeScript

**メリット**:
- エコシステムが最大（UIライブラリ、ツール）
- 採用実績が豊富、情報が多い
- チームに経験者が3名

**デメリット**:
- 状態管理の選択肢が多く、決定が必要
- ボイラープレートが多い傾向

### 選択肢2: Vue 3 + TypeScript

**メリット**:
- 公式ツールが統一されている
- 学習曲線がやや緩やか
- Composition APIでReactに近い書き方も可能

**デメリット**:
- チームに経験者が1名
- エコシステムがReactより小さい

---

## 決定

React + TypeScriptを採用する。

### 理由

1. チームの既存スキルを活かせる（経験者3名 vs 1名）
2. 必要なUIコンポーネント（データテーブル、チャート）のライブラリが豊富
3. 長期的なメンテナンス性（採用実績、コミュニティの大きさ）

---

## 影響

- 状態管理はZustandを採用（別途ADR-0003で決定）
- UIライブラリはshadcn/uiを採用
```

## 良い例3: 認証方式の選定（シンプルテンプレート）

小規模〜中規模の決定をシンプルに記録した例。

```markdown
# ADR-0003: JWT認証を採用

## ステータス

承認

## 日付

2024-01-25

## コンテキスト

APIの認証方式を決定する。SPAとモバイルアプリの両方から利用される。

## 決定

JWTベースの認証を採用する。

## 理由

- ステートレスでスケーラブル（サーバー側にセッション情報を持たない）
- SPAとモバイルアプリで統一的に利用可能
- リフレッシュトークンでセッション管理が可能

## 他に検討した選択肢

- **セッション認証**: サーバー側でセッション管理が必要になり、スケールアウト時に複雑化する
- **OAuth 2.0のみ**: 自前サービス間の認証には過剰。外部連携時に追加で対応すればよい

## 影響

- トークンの有効期限: アクセストークン15分、リフレッシュトークン7日
- リフレッシュトークンはHttpOnly Cookieに保存
- トークン無効化のためにブラックリスト機能を実装（Redis使用）
```

## 悪い例と改善

### 悪い例1: 選択肢が1つだけ

```markdown
# ADR-0001: PostgreSQLを使う

## コンテキスト

データベースが必要。

## 決定

PostgreSQLを使う。良いデータベースだから。

## 影響

なし。
```

**問題点:**
- 選択肢の比較がない — なぜ他のDBではないのか分からない
- 「良いから」では理由にならない — プロジェクト固有の根拠がない
- 影響が「なし」 — 技術決定には必ず何らかの影響がある
- 背景が「データベースが必要」だけ — どんなデータをどう使うのか不明

**改善版:**

```markdown
# ADR-0001: PostgreSQLをメインデータベースとして採用

## ステータス

承認

## 日付

2024-01-15

## コンテキスト

社内の在庫管理システムを新規開発する。商品マスタ、在庫数、
入出庫履歴を管理する。データ整合性が重要で、月次レポートの
生成にも使用する。

## 検討した選択肢

### PostgreSQL
- メリット: ACID準拠、複雑なクエリに強い、チームに経験者あり
- デメリット: 水平スケーリングが難しい

### MySQL
- メリット: シンプル、実績豊富
- デメリット: Window関数のサポートが限定的、JSON操作が弱い

## 決定

PostgreSQLを採用する。月次レポートでWindow関数やCTEを多用する
見込みがあり、その点でMySQLより適している。チームにPostgreSQL
経験者が2名いることも決め手。

## 影響

- AWS RDS PostgreSQLを使用（月額コスト約$XX）
- 水平スケーリングが必要になった場合はリードレプリカで対応
```

### 悪い例2: メリットだけ/デメリットだけ

```markdown
## 検討した選択肢

### PostgreSQL

**メリット**:
- 高性能
- 無料
- 実績豊富

### MongoDB

**デメリット**:
- トランザクションがない
- 複雑
- 学習コストが高い
```

**問題点:**
- PostgreSQLのデメリットがない — 完璧な技術はない
- MongoDBのメリットがない — 選択肢として挙げた意味がない
- 結論ありきの偏った記述 — 後から読んだ人が判断の公平性を信頼できない

**改善版:**

```markdown
## 検討した選択肢

### PostgreSQL

**メリット**:
- ACID準拠で整合性が高い
- 複雑なクエリ（Window関数、CTE）に強い
- OSSで無料、マネージドサービスも安価

**デメリット**:
- 水平スケーリングは自前での対応が必要
- 大量の書き込みではNoSQLに劣る

### MongoDB

**メリット**:
- スキーマレスで開発初期の変更に強い
- 水平スケーリングが容易
- 書き込みスループットが高い

**デメリット**:
- 複数コレクション間のトランザクションが限定的
- 複雑な結合クエリが苦手
- チームに経験者がいない
```

### 悪い例3: 曖昧な理由

```markdown
## 決定

Reactを採用する。

### 理由

- モダンだから
- みんな使っているから
- 良さそうだから
```

**問題点:**
- 「モダン」「良さそう」は具体性がない — 何がモダンで、何が良いのか不明
- プロジェクト固有の理由がない — どんなプロジェクトでも同じことが言える
- 再現性がない — 同じ状況で同じ結論に至るか判断できない

**改善版:**

```markdown
## 決定

Reactを採用する。

### 理由

1. **チームスキル**: 5名中3名にReact実務経験があり、立ち上げが速い
2. **UIコンポーネント**: 管理画面に必要なデータテーブル・チャートライブラリが
   Reactエコシステムに最も充実している（Tanstack Table、Rechartsなど）
3. **採用・教育**: React経験者の採用市場が大きく、新メンバーの教育コストも低い
```

## 良いADRのチェックリスト

- [ ] 背景と課題が具体的に説明されているか（プロジェクト固有の文脈があるか）
- [ ] 最低2つの選択肢を検討しているか
- [ ] 各選択肢のメリット・デメリットが公平に記載されているか
- [ ] 決定理由が背景・課題と論理的に繋がっているか
- [ ] ネガティブな影響も記載されているか（可能なら軽減策も）
- [ ] 第三者が読んで理解・納得できるか
- [ ] 1年後に読んで「なぜこの決定をしたか」が分かるか
