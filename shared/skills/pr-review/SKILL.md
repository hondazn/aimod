# PRレビュー（汎用）

## ユーザー入力

```text
$ARGUMENTS
```

作業を開始する前に、ユーザーからの入力を理解し、PR番号を特定すること。

## 目的

PRの変更内容を構造的に理解・解説し、一般的なコード品質観点でレビューし、GitHubにレビューコメントを投稿する。
このスキルはコード修正を行わない。

---

## Phase 1: PR情報取得

### 1-1. PR番号の抽出

`$ARGUMENTS` から以下の形式でPR番号を抽出する:

- `#123` / `123` — 数字のみ
- `https://github.com/{owner}/{repo}/pull/123` — URLからパース
- 引数なし — カレントブランチのPRを自動検出: `gh pr view [<number> | <url> | <branch>] --json number --jq '.number'`

抽出できない場合はユーザーにPR番号を確認する。

### 1-2. PR情報の取得

```bash
gh pr view <番号> --json number,title,state,headRefName,baseRefName,url,body,labels,author,reviewRequests
```

エラーが発生した場合（PRが存在しない、権限不足等）はユーザーに報告して停止する。

### 1-3. 前提条件チェック

| チェック項目 | 条件 | 失敗時 |
|-------------|------|--------|
| PRステータス | `state == OPEN` | MERGED/CLOSEDなら「このPRは既に{状態}です」と報告して停止 |

### 1-4. リポジトリのオーナー/リポジトリ名の取得

以降の `gh api` コマンドで使うため、最初にオーナー/リポジトリ名を取得しておく:

!`gh repo view --json nameWithOwner --jq '.nameWithOwner'`

取得した値（例: `spiderplus/SP-web-ui`）は以降の `{owner}/{repo}` プレースホルダーにリテラルとして埋め込む。

### 1-5. 既存レビューの確認

自分（CLIユーザー）が既にこのPRをレビュー済みかを確認する:

自分のログイン名を取得
!`gh api user --jq '.login'`

```bash
# このPRの全レビューを取得
gh api repos/{owner}/{repo}/pulls/<番号>/reviews \
  --jq '[.[] | {user: .user.login, state, body, submitted_at: .submitted_at}]'
```

取得した結果から、自分のログイン名と一致するレビューを抽出する。

**再レビューの場合（自分のレビューが1件以上存在する場合）:**

- `is_re_review = true` とし、最新の自分のレビューから以下を記録する:
  - `previous_review_state`: `CHANGES_REQUESTED` / `COMMENTED` / `APPROVED`
  - `previous_review_body`: レビューサマリー本文
- これらの情報は Phase 1-5A の再レビューガード、Phase 4 の一貫性チェック、Phase 5-3 のサマリー作成で使用する

**初回レビューの場合:** `is_re_review = false` として続行する。

### 1-5A. 再レビューガード

`is_re_review == true` の場合、以下の順序でチェックする。

**ステップ1: APPROVE後ガード**

| 前回の状態 (`previous_review_state`) | デフォルト動作 |
|--------------------------------------|---------------|
| `APPROVED` | **停止**。「このPRは既にAPPROVEしています。再レビューは投稿しません。」と報告して終了する。ユーザーの引数（`$ARGUMENTS`）に「再レビュー」「もう一度」「re-review」等の明示的な再レビュー指示が含まれる場合に限り、`post_approval_mode = true` として続行する |
| `CHANGES_REQUESTED` | 続行 |
| `COMMENTED` | 続行 |

**ステップ2: レビューラウンド管理**

Phase 1-5 で取得した自分のレビュー一覧から、レビュー回数（`review_count`）を算出する。

| review_count | 動作 |
|---|---|
| 0 | 初回レビュー。通常続行 |
| 1 | 再レビュー。Phase 1-5B へ進む |
| 2以上 | ユーザーに確認: 「このPRに既に{review_count}回レビューしています。PRオーサーへの負担を考慮し、追加レビューが本当に必要か確認させてください。」ユーザーの続行指示がなければ停止する |

### 1-6. 関連Issueの取得

PR本文から `Closes #N` / `Fixes #N` / `Resolves #N` / `Refs #N` / `#N` パターンでIssue番号を抽出し、Issue情報を取得する:

```bash
gh issue view <番号> --json number,title,body,labels
```

Issueが見つからない場合は無視して続行する（エラーで止めない）。

### 1-7. 変更ファイル一覧と差分の取得

```bash
gh pr view <番号> --json files --jq '.files[] | "\(.path)\t\(.additions)\t\(.deletions)"'
gh pr diff <番号>
```

### 1-8. 大規模PR対応

変更ファイル数が30超、または差分行数合計が2000行超の場合、以下の優先順位で重点レビュー対象を自動選択する:
   - ビジネスロジック（テスト・設定ファイル・自動生成ファイルより優先）
   - 変更量の大きいファイル
   - 新規追加ファイル

### 1-9. 前回コメントと既存コメントの収集（再レビュー時のみ）

`is_re_review == true` の場合、二転三転の防止と重複排除のためにPR上の既存レビューコメントを収集する。

respond-review スキルと同じ GraphQL API でレビュースレッドを取得する:

```bash
gh api graphql -f query='
query($owner: String!, $repo: String!, $number: Int!) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $number) {
      reviewThreads(first: 100) {
        nodes {
          isResolved
          isOutdated
          path
          line
          comments(first: 50) {
            nodes { body, author { login }, createdAt }
          }
        }
      }
    }
  }
}' -f owner='{owner}' -f repo='{repo}' -F number=<PR番号>
```

取得結果を2つに分類して保持する:

- **`my_previous_comments`**: 自分のログイン名と一致するコメント。Phase 4 の一貫性チェックで使用する
- **`other_comments`**: 自分以外（bot含む）のコメントで `isResolved == false` かつ `isOutdated == false` のもの。Phase 4-5 の重複排除で使用する

---

## Phase 2: コンテキスト理解

差分だけでなく、変更の文脈を把握するために周辺コードを読む。

### 2-1. 周辺コードの読み込み

変更ファイルのうち、特に以下のケースでは周辺コードをファイル読み込みで確認する:

- 関数シグネチャの変更 — 呼び出し元を確認
- 型定義の変更 — 使用箇所を確認
- インターフェース/トレイトの変更 — 実装箇所を確認
- インポートの追加 — 依存先のAPIを確認

全ファイルを読む必要はない。差分から判断して、レビューに必要な範囲だけ読む。

### 2-2. プロジェクト規約の確認

以下が存在するか確認し、あれば内容を把握する:

- AIアシスタント向け設定ファイル（`CLAUDE.md`, `.cursorrules` 等） — プロジェクトのコーディング規約
- `CONTRIBUTING.md` — コントリビューションルール
- PRテンプレート（`.github/pull_request_template.md`）

### 2-3. プロジェクト固有レビューエージェントの確認

汎用エージェント（correctness-reviewer, security-perf-reviewer, quality-test-reviewer）やプラグインエージェント（pr-review-toolkit:* 等）以外に、プロジェクト固有のレビューエージェントがないか確認する。

**方法1（優先）: subagent_type から検出**

利用可能な subagent_type の中に、ファイル名に `reviewer` を含むプロジェクト固有エージェントがないか確認する。汎用3エージェントとプラグインエージェントは除外する。

**方法2（フォールバック）: ファイル名ベースの探索**

方法1で見つからない場合、`.claude/agents/*reviewer*.md` にマッチするファイルを Glob で検索する。ヒットしたファイルの frontmatter を Read して、除外条件（description に `respond-review用` または `Agent Teams用` を含む）に該当しないものを採用する。

検出した場合は「プロジェクト固有レビューエージェントを検出: <名前一覧>」と表示する。0件の場合は何も表示せず続行する。

---

## Phase 3: 変更内容の構造的解説

このPhaseではレビュー（良し悪しの判断）は行わない。純粋に「何が変更されたか」を解説する。

### 3-1. 変更内容の解説

- 追加/変更/削除されたファイルの一覧（各ファイルに変更主旨を1行添える）
- 追加/変更/削除された型・関数・クラスの一覧
- 各変更の目的と役割

### 3-2. 設計意図の推論

PR差分・本文・関連Issueから以下を推論する:

- **解決する課題**: このPRが何を解決しようとしているか
- **実装アプローチ**: 選択された方法とその理由
- **スコープの判断**: 含むもの・含まないもの

不確かな点は「〜と読めますが、意図が違ったら教えてください」のように確認を促す。

### 3-3. 影響分析

- **直接影響**: 変更されたファイルが直接依存している/されているコード
- **間接影響**: API変更によるクライアントへの影響、スキーマ変更等

### 3-4. レビュアー向けの着眼点

PRの文脈に即した具体的な質問を3〜5個、オープンクエスチョン形式で提示する。

---

## Phase 4: コードレビュー

変更されたファイルのみを対象にレビューする。Phase 2で確認したプロジェクト規約があれば、それも加味する。

レビューの心構え: 本当に意味のある問題だけを指摘する。些末な問題を大量に指摘するより、重要な問題を正確に指摘するほうがレビュイーにとって価値がある。「もし自分がこのコードの動作に責任を持つとしたら、何が気になるか」という観点で考える。

**再レビュー時のルール（`is_re_review == true` の場合）:**

再レビューでは、全体のdiffを引き続きコンテキストとして読むが、findings の投稿基準を引き上げる。PRオーサーは既に前回のフィードバックに対応する努力をしており、同じ箇所で繰り返し指摘されることは技術的問題以上に心理的ダメージが大きい。

重要度閾値:

| モード | GitHub投稿対象 | 抑制（ユーザー報告のみ） |
|---|---|---|
| 初回レビュー | must, suggestion, nit, good | なし |
| 再レビュー | must, suggestion | nit, good |
| APPROVE後再レビュー (`post_approval_mode`) | must のみ | suggestion, nit, good |

`good`（良い実装への称賛）は初回レビューでのみ投稿する。再レビューで改めて褒めると冗長になる。

抑制された findings は GitHub に投稿しない。ユーザーには「抑制: N件」として件数のみ報告する。

一貫性ルール（二転三転防止）:

再レビュー時は Phase 1-9 で取得した `my_previous_comments` を参照し、以下の原則を守る:

1. **既に指摘済みの問題は再指摘しない** — 同じファイル + 行番号差5行以内で、同種の問題を指摘済みの場合はスキップする
2. **前回の指摘と矛盾する指摘を出さない** — 前回「Aにすべき」と指摘した箇所で「Aにすべきでない」と指摘することは禁止。方針を変えたい場合は GitHub に投稿せず、ユーザーに「前回の指摘と矛盾する可能性があります」と相談する
3. **前回指摘が対応された箇所への新規指摘は慎重に** — PRオーサーが前回のフィードバックに従って修正した結果に対し、さらに別の問題を指摘する場合は重要度 `must` の場合のみ。`suggestion` / `nit` レベルの追い打ちはしない
4. **対応確認は暗黙に行う** — 前回の CHANGES_REQUESTED 指摘が修正されたかの確認は、レビュープロセスの中で自然に行う。個別の「修正確認しました」コメントをわざわざ投稿する必要はない（サマリーで全体的に触れれば十分）

### 4-1. レビュー方式の決定

PRの規模に応じてレビュー方式を選択する:

- **差分200行以下** かつ **変更ファイル5個以下** → 4-2（単一フローレビュー）へ
- **それ以外** → 4-3（サブエージェント並列レビュー）へ

この判定はPhase 1-7で取得した差分行数・ファイル数から行う。

**プロジェクト固有エージェントの扱い:**

Phase 2-3 で検出されたプロジェクト固有エージェントがある場合:

- **小規模PR（4-2）**: 単一フローレビュー完了後、4-3A でプロジェクト固有エージェントのみ追加起動する
- **大規模PR（4-3）**: 汎用3エージェントとプロジェクト固有エージェントを 4-3 + 4-3A で同一メッセージ内で同時に並列起動する

### 4-2. 単一フローレビュー（小規模PR用）

以下の5観点で確認する。全観点を機械的にチェックするのではなく、PRの文脈に合った観点に集中する。

**正しさ・堅牢性**
- ロジックの誤り、境界条件の見落とし
- エラーハンドリングの漏れ（パニック、unwrap、空catch等）
- null/undefined の未考慮
- 競合状態やデッドロックの可能性

**セキュリティ**
- 入力バリデーション不足
- インジェクション系脆弱性（SQL, XSS, コマンドインジェクション等）
- 機密情報のハードコード
- 認証・認可チェックの漏れ

**テスト**
- 新規関数/メソッドにテストがあるか
- 変更された関数のテストが更新されているか
- 意味のあるアサーションがあるか

**パフォーマンス**
- N+1クエリ、不要なループ、大量データの非効率な処理
- メモリリークの可能性

**可読性・保守性**
- 不明瞭な命名
- 不要な複雑さ、重複コード
- デッドコード

検出結果は 4-7 の表形式で整理する。

### 4-3. サブエージェント並列レビュー

3つの専門エージェントを並列起動し、異なる観点でレビューする。各エージェントが独立した専門家の視点を持つことで、単一フローでは見落としがちな問題をより深く検出できる。

以下の3つのエージェントを **可能な限り同時に** 起動する:

| エージェント名 | 専門性 | 担当観点 |
|---|---|---|
| `correctness-reviewer` | 堅牢なシステムを作るシニアエンジニア | 正しさ・堅牢性 |
| `security-perf-reviewer` | セキュリティとパフォーマンスの専門家 | セキュリティ + パフォーマンス |
| `quality-test-reviewer` | テスト駆動開発を重視するコード品質エキスパート | テスト + 可読性・保守性 |

各エージェントの起動パラメータ:

- `name`: 上記エージェント名（`agents/` ディレクトリに定義済み）
- `prompt`: Phase 1-3 で取得した PR コンテキストを渡す。以下の情報を含めること:
  1. PR情報（番号、タイトル、ブランチ、作成者）
  2. PR本文
  3. 関連Issue情報
  4. Phase 3 の設計意図推論結果
  5. 変更ファイル一覧と各ファイルの変更主旨
  6. `gh pr diff` の出力（差分全文）
  7. プロジェクト規約（Phase 2-2 で確認した内容）
  8. **再レビュー時のみ**: `my_previous_comments` の内容（前回自分が指摘したファイル・行・内容の一覧）と以下の指示を追記する:
     - 「これは再レビューです。前回の指摘と矛盾する新しい指摘は出さないでください。既に指摘済みの問題は再指摘しないでください。」
     - `post_approval_mode` 時は追加で: 「APPROVE後の再レビューです。must レベル（正しく動作しない、セキュリティリスク）の問題のみ報告してください。」

### 4-3A. プロジェクト固有エージェント起動

Phase 2-3 で検出されたプロジェクト固有エージェントがある場合のみ実行する。

**方法1（subagent_type で起動可能な場合）:**

Agent ツールの `subagent_type` パラメータにエージェント名を指定して起動する。prompt には Phase 1-3 で取得した PR コンテキスト（PR情報、変更ファイル一覧、差分全文、プロジェクト規約）を渡す。

**方法2（フォールバック: Read + prompt 埋め込み）:**

subagent_type で利用できない場合:
1. エージェントの `.md` ファイルを Read で全文読み込む
2. YAML frontmatter を除いた本文をプロンプトのベースとする
3. PR コンテキストをプロンプト末尾に付加して Agent ツールで起動する

**起動タイミング:**

- 大規模PR: 汎用3エージェント（4-3）と**同一メッセージ内で同時に**起動する
- 小規模PR: 4-2 完了後に単独で起動する

### 4-4. 結果の受信

汎用サブエージェントとプロジェクト固有エージェントの全結果を待つ。各エージェントは JSON 形式で `findings` 配列を返す。

**汎用エージェントの結果:** JSON `findings` 配列（既存の処理）

**プロジェクト固有エージェントの結果:**

1. JSON `findings` 配列を含む → そのまま処理
2. テーブル形式 → 各行を finding に変換する:
   - `ファイル:行` → `file` + `line`
   - `問題の内容` → `body`
   - 重要度: エージェントの出力ラベルを本スキルの4段階（`must` / `suggestion` / `nit` / `good`）に正規化する。一致しないラベルは `suggestion` にフォールバックする
   - `source` フィールドを付与: `"<agent-name> (project)"`
3. その他テキスト → findings を手動で抽出して構造化する

パース失敗時の対処:
- 抽出も困難な場合、そのエージェントの結果は除外し、残りの結果で続行する

### 4-5. 結果統合

全エージェント（汎用 + プロジェクト固有）の findings を統合する。

**重複検出ルール:**

1. **完全重複**（同一ファイル + 行番号差5行以内 + 内容が実質同一）→ 重要度が高い方を採用
2. **部分重複**（同一ファイル + 行番号差5行以内 + 異なる観点からの指摘）→ コメント本文を統合して1コメントにまとめる。重要度ラベルは最も高いものを適用
3. **クロスソース重複**（汎用 + プロジェクト固有が同一ファイル + 行番号差5行以内 + 同種の指摘）→ プロジェクト固有の指摘を優先（プロジェクト文脈をより深く理解しているため）。コメント本文は統合する
4. **非重複** → そのまま採用
5. **自分の前回コメントとの重複**（再レビュー時のみ。`my_previous_comments` と同一ファイル + 行番号差5行以内 + 内容が実質同一）→ 除外する
6. **他レビュアーのコメントとの重複**（再レビュー時のみ。`other_comments` と同一ファイル + 行番号差5行以内 + 内容が実質同一）→ 除外する。ただし `isResolved == true` のスレッドは除外対象にしない（解決済み問題の再発を検出するため）
7. **前回指摘との矛盾チェック**（再レビュー時のみ）— finding が `my_previous_comments` の指摘と矛盾する場合（例: 前回「Aを使うべき」→ 今回「Aを使うべきでない」）→ GitHub投稿対象から除外し、ユーザーに「前回の指摘と矛盾する可能性があります」と報告する

**ソート:** must → suggestion → nit → good の順に並べる。同一重要度内ではファイルパス順。

### 4-6. レビュー観点（参考）

サブエージェントが担当する5つの観点の全体像:

| 観点 | 担当エージェント | 主な検出対象 |
|------|----------------|-------------|
| 正しさ・堅牢性 | correctness-reviewer | ロジック誤り、境界条件、エラーハンドリング、null安全性、競合状態 |
| セキュリティ | security-perf-reviewer | 入力バリデーション、インジェクション、機密情報、認証・認可 |
| パフォーマンス | security-perf-reviewer | N+1クエリ、メモリリーク、非効率処理 |
| テスト | quality-test-reviewer | テスト有無、アサーション品質、エッジケース |
| 可読性・保守性 | quality-test-reviewer | 命名、複雑さ、重複、デッドコード |

### 4-7. 検出結果の整理

単一フロー（4-2）またはサブエージェント統合（4-5）の結果を以下の表形式で整理する。

**プロジェクト固有エージェントが起動された場合**（ソース列を追加）:

```text
| # | ファイル:行 | 問題の内容 | 重要度 | 観点 | ソース |
|---|-----------|-----------|--------|------|--------|
| 1 | src/foo.rs:42 | nullチェックが抜けていて、undefinedアクセスでクラッシュする | must | 正しさ | correctness-reviewer |
| 2 | crates/app/src/bar.rs | 仕様変更にテストが含まれていない | must | テスト | spec-reviewer (project) |
| 3 | src/baz.rs:15 | typo: recieve → receive | nit | 可読性 | quality-test-reviewer |
```

**プロジェクト固有エージェントがない場合**（従来形式）:

```text
| # | ファイル:行 | 問題の内容 | 重要度 | 観点 |
|---|-----------|-----------|--------|------|
| 1 | src/foo.rs:42 | nullチェックが抜けていて、undefinedアクセスでクラッシュする | must | 正しさ |
| 2 | src/bar.rs:15 | typo: recieve → receive | nit | 可読性 |
```

問題が0件の場合はその旨を報告し、Phase 5でAPPROVEコメントのみ作成する。

## Phase 5: コメント作成

### 5-1. 重要度分類

| 重要度 | 意味 | 例 |
|--------|------|-----|
| **must** | 正しく動作しない、セキュリティリスク | nullアクセスでクラッシュ、SQLインジェクション |
| **suggestion** | より良い実装が存在する | エラーメッセージの改善、リファクタリング提案 |
| **nit** | 些細な改善点 | typo、スタイルの統一 |
| **good** | 良い実装、学びになるパターン | 巧みなエラーハンドリング、良い抽象化 |

### 5-2. インラインコメントの書き方

- ですます調で書く
- 同僚レビュアーのような自然な表現を使う
  - 良い例: 「ここ、nullが来るとクラッシュしそうです。チェック入れたほうが安全かも」
  - 悪い例: 「null参照の可能性が検出されました。適切なバリデーションの実装が推奨されます」
- コメントの先頭に重要度バッジを付け、バッジの後に改行を入れる: `![must](https://img.shields.io/badge/review-must-red.svg)` `![suggestion](https://img.shields.io/badge/review-suggestion-blue.svg)` `![nit](https://img.shields.io/badge/review-nit-green.svg)` `![good](https://img.shields.io/badge/review-good-brightgreen.svg)`
- 問題を指摘するだけでなく、可能であれば改善案を添える
- たまに「!」や絵文字を使って、人間らしい温かみを出す
  - 頻度: コメント3〜4件に1回程度。毎回使うと逆に不自然なので、使わないコメントのほうが多くていい
  - 「!」は肯定的な文脈で使う（称賛、感謝、同意）。問題指摘では使わない
  - 絵文字は文末に1つだけ添える。以下から選ぶ:
    - 👀 注目してほしい箇所
    - 👍 良い実装への賛同
    - 🎉 LGTM・称賛
    - ⚠️ 潜在的リスクへの注意喚起
    - 💡 提案・アイデア
    - 🙏 感謝
  - 良い例: 「このエラーハンドリング、丁寧でいいですね 👍」
  - 悪い例: 「ここ、nullが来るとクラッシュしそうです! 😱💥 チェック入れたほうが安全かも!! ⚠️🙏」
- サマリーとインラインコメントは別物。サマリーはPR全体の印象を伝える場で、個別の指摘内容を繰り返す場ではない

### 5-3. レビューサマリー

レビューサマリーは GitHub Pull Request Review の `body` として投稿される。同僚レビュアーがPRに残すコメントと同じトーンで書く。

**サマリーの役割:**

サマリーはPR全体に対する印象・評価を伝える場であり、個別の指摘を伝える場ではない。個別の指摘はすべてインラインコメントが担う。読み手はサマリーの直後にインラインコメントを見るため、サマリーでインラインの内容に言及する必要はない。

**構成ルール:**

1. 冒頭に全体印象を一文で書く — mustがあれば何が問題かを端的に、なければ肯定的な印象から入る
2. mustがある場合は問題の **領域** に触れてよい（例: 「エラーハンドリング周りに気になるところがあります」）。ただし具体的な指摘内容（どの関数で何が起きているか等）はインラインに任せ、サマリーでは繰り返さない
3. 設計方針やアーキテクチャについて議論が必要な場合のみ、補足を書く
4. 件数の統計表は書かない
5. Markdownの見出し（##, ###）は使わない
6. 「インラインで書きました」「詳細はインラインを見てください」等、インラインコメントの存在に言及しない。GitHubのUIで自明であり、情報量がない
7. 全体で1-4行に収める
8. **再レビューの場合**（`is_re_review == true`）は1-2行に収める。mustがなければ「LGTM」等の一言で十分。「前回のレビューでは〜」のような冗長な振り返りはしない

**バリエーションの原則:**

サマリーを毎回同じ型で書かない。以下の4つの「文の型」を意識し、直近のレビューと異なる型を使う:

1. **印象から入る** — 全体の印象を述べてから補足。例: 「読みやすい実装ですね。」
2. **変更の核心から入る** — PRの主題に直接触れる。例: 「認証フローの刷新、すっきりまとまっていると思います。」
3. **端的に評価する** — 一言で済ませる。例: 「いい感じです!」「LGTM 🎉」
4. **気になる点から入る** — mustがある場合、ポジティブを省いて本題に入る。例: 「エラーハンドリングで1点気になるところがあります。」

型の選び方: PRの内容・規模に応じて自然な型を選ぶ。ただし「印象から入る + ただ、」の型を連続で使うのは避けること。

**再レビュー時のサマリー補正:**

`is_re_review == true` の場合、冒頭の一文を以下の方針で書く:

| 前回の状態 (`previous_review_state`) | 今回の結果 | 冒頭のトーン |
|--------------------------------------|-----------|-------------|
| `CHANGES_REQUESTED` | mustなし | 前回の指摘が対応されたことを認めつつ、肯定的に |
| `CHANGES_REQUESTED` | mustあり | 前回に続き気になる点がある旨を端的に |
| `COMMENTED` | mustなし | 引き続き良い感じである旨を |
| `COMMENTED` | mustあり | 新しく気になった点がある旨を |
| `APPROVED` | mustあり | APPROVEした後に気づいた点がある旨を丁寧に |
| `APPROVED` | mustなし | （GitHub に投稿しない） |

**再レビューサマリー例:**

再レビューのサマリーは初回より短く、あっさりと書く。PRオーサーは既に文脈を理解しているので、冗長な説明は不要。mustがなければ1行で済ませる。

前回 CHANGES_REQUESTED → 今回 mustなし:
> LGTM、修正ありがとうございます!

前回 CHANGES_REQUESTED → 今回 mustあり:
> 修正ありがとうございます。1点だけ気になるところがあります。

前回 COMMENTED → 今回 mustなし:
> LGTM 🎉

前回 COMMENTED → 今回 mustあり:
> 1点だけ気になったところがあります。

前回 APPROVED → 今回 mustあり（`post_approval_mode`）:
> APPROVE後に気づいたのですが、1点だけ確認させてください。

初回レビュー（`is_re_review == false`）は従来通りのサマリー例を使う。

**再レビュー時の心理的安全性:**

再レビューはPRオーサーにプレッシャーを与えうる。以下を守ること:

- `post_approval_mode` で must が0件の場合、GitHub にはレビューを**投稿しない**。ユーザーにのみ「問題なし」を報告する
- 再レビューでの指摘は最大3件に絞る。それ以上ある場合はユーザーに「{N}件の指摘がありますが、すべて投稿しますか？」と確認する
- `post_approval_mode` では `REQUEST_CHANGES` を使わない（`COMMENT` のみ）。一度 APPROVE したものに CHANGES_REQUESTED を出すのは混乱を招く

**文体ルール:**

| ルール | NG | OK |
|--------|-----|-----|
| 報告書調にしない | 「以下にレビュー結果を記載します」 | （書かない。本題から始める） |
| 冗長な前置きを入れない | 「PRの変更内容を確認しました。全体として〜」 | 「設計・実装ともに良さそうです。」 |
| 件数で語らない | 「must: 2件を検出しました」 | 「2点ほど直したほうがよさそうなところがあります」 |
| テンプレート的な締めを入れない | 「以上がレビュー結果です。ご確認ください。」 | （書かない。最後の指摘で終わる） |
| 敬語は軽めに | 「ご修正いただけますと幸いです」 | 「直してもらえると助かります」 |
| 曖昧にしない | 「問題がある可能性が考えられます」 | 「ここ、バグになりそうです」 |
| インラインに言及しない | 「細かい点はインラインで書きました」 | （書かない。自明なので不要） |
| 絵文字・!は控えめに | 「LGTM 🎉👍✨」 | 「LGTM 🎉」 |

**サマリー例:**

型ごとに複数の例を示す。これらは参考であり、コピーして使い回す対象ではない。PRの文脈に合わせて自分の言葉で書くこと。

印象から入る:
> 読みやすい実装ですね。テストも丁寧で助かります。
> シンプルにまとまっていて良いと思います。ただ、バリデーション周りで少し気になるところがあります。

変更の核心から入る:
> キャッシュ戦略の見直し、良い判断だと思います。
> API設計の変更、方向性は良さそうです。エラーレスポンスの扱いだけ確認させてください。

端的に評価する:
> いい感じです!
> LGTM 🎉

気になる点から入る:
> 並行処理周りで修正が必要そうなところがあります。
> 1点だけ、型安全性で気になるところがあります。

### 5-4. 校正チェック

サマリー作成後、投稿前に確認する:

- 「## レビューサマリー」のような見出しで始めていないか
- 「非常に」「極めて」等の過剰修飾がないか
- 4行を超えていないか
- 「インラインで書きました」「詳細はインラインを〜」等、インラインコメントの存在に言及していないか
- インラインコメントと同じ具体的指摘（関数名、変数名、具体的な問題の説明）をサマリーで繰り返していないか
- 再レビュー（`is_re_review == true`）なのに初回レビューのような文面（例: 「設計の方向性は良いと思います」のような初見の印象）になっていないか
- 初回レビュー（`is_re_review == false`）なのに「前回」に言及していないか
- 「!」や絵文字が多用されていないか（1つのサマリーに絵文字は最大1つ、「!」は最大1つ）
- 同僚に口頭で伝えるとしたら不自然でないか
- 直近3回のレビューサマリーと同じ文の型（同じ文頭パターン、同じ接続構造）を繰り返していないか

## Phase 6: GitHub投稿

### 6-1. レビューイベントの決定

**初回レビュー時:**

| 条件 | レビューイベント |
|------|----------------|
| must が1件以上 | `REQUEST_CHANGES` |
| suggestion/nit のみ | `COMMENT` |
| good のみ / 問題なし | `APPROVE` |

**再レビュー時（以下が通常ルールより優先）:**

| 条件 | レビューイベント |
|------|----------------|
| `post_approval_mode` + mustなし | **投稿しない**（ユーザーにのみ報告） |
| `post_approval_mode` + mustあり | `COMMENT`（`REQUEST_CHANGES` にしない） |
| 再レビュー（上記以外） | 通常ルール通り |

### 6-2. コメントJSONの構築と投稿

まずHEAD commit SHAを取得する:

```bash
gh pr view <番号> --json headRefOid --jq '.headRefOid'
```

heredocで直接APIに投稿する:

```bash
gh api repos/{owner}/{repo}/pulls/<番号>/reviews \
  --method POST \
  --input - <<'EOF'
{
  "event": "COMMENT",
  "body": "レビューサマリー本文",
  "commit_id": "<HEAD SHA>",
  "comments": [
    {
      "path": "src/xxx.rs",
      "line": 42,
      "side": "RIGHT",
      "body": "![must](https://img.shields.io/badge/review-must-red.svg)\nコメント内容"
    }
  ]
}
EOF
```

注意点:
- `line` はPR diff上で表示される行番号（変更後のファイルの行番号）を使う
- `side` は原則 `"RIGHT"`（変更後側）を指定する。削除行にコメントする場合のみ `"LEFT"`
- 複数行にまたがるコメントが必要な場合は `start_line` と `start_side` も指定する
- heredocのデリミタを `<<'EOF'` （クォート付き）にすることでシェル変数展開を防ぐ。動的な値（commit SHA等）はリテラルとして埋め込むこと

**エラー時の対応:**
- 422 Unprocessable Entity（行番号がdiff範囲外等）— 問題のコメントを除外して再試行する。除外したコメントはユーザーに報告する
- 403 Forbidden — 権限不足を報告して停止する
- その他のエラー — エラー内容をユーザーに報告して停止する

### 6-3. 完了報告

**初回レビュー時:**

```text
レビューを投稿しました

- PR: #<番号> <タイトル>
- イベント: COMMENT / REQUEST_CHANGES / APPROVE
- コメント数: N件（must: X, suggestion: Y, nit: Z, good: W）
- URL: <PR URL>
```

**再レビュー時:**

```text
レビューを投稿しました

- PR: #<番号> <タイトル>
- イベント: COMMENT
- 投稿: N件（must: X, suggestion: Y）
- 抑制: M件（既存指摘との重複: A件、閾値未満: B件）
- レビューラウンド: 第{review_count+1}回
- URL: <PR URL>
```

**`post_approval_mode` + mustなし:**

```text
再確認しましたが、新しい変更に問題はありませんでした。
APPROVEのままで問題ないと思います。GitHubへの追加コメントは投稿しませんでした。

- PR: #<番号> <タイトル>
- URL: <PR URL>
```

投稿に一部失敗があった場合は、成功したコメントと失敗したコメントを分けて報告する。
