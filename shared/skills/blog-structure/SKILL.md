# Structural Thinking - ブログ記事の構造設計

## ユーザー入力

```text
$ARGUMENTS
```

## 設計プロセス

以下のステップに沿って、ユーザーの入力から記事構成を組み立てる。

### Step 1: 素材の把握と整理

まずユーザーの入力から以下を読み取る。不明な点があれば推測せず確認する。

- **テーマ**: 何について書くのか
- **主張・結論**: 最終的に読者に伝えたいこと
- **想定読者**: 誰に向けた記事か（技術レベル、前提知識、関心）
- **記事の種類**: 技術解説、体験記、比較記事、チュートリアル、意見表明など
- **素材**: ユーザーが持っているアイデアの断片、メモ、キーワード

情報が十分であればそのまま進む。不足している場合は、最小限の質問で補う。
ただし質問攻めにはしない。推測可能な部分は仮置きし、構成案に「（要確認）」と注記する。

### Step 2: 読者の思考フローを設計する

記事構成の本質は「読者の頭の中にどういう順番で情報を届けるか」の設計にある。
以下の観点で読者の思考フローを考える。

- **読者の出発点**: 記事を読む前に読者が知っていること・感じていること
- **読者のゴール**: 記事を読み終えた後に読者がどうなっていてほしいか
- **ギャップ**: 出発点からゴールに至るまでに埋めるべき知識・感情の差分
- **順序**: ギャップを自然に埋めていく情報提示の順番

### Step 3: 構成パターンの選択

テーマと記事の種類に合った構成パターンを選ぶ。以下は代表的なパターンだが、組み合わせや変形も可。

| パターン | 適する場面 | 流れ |
|---------|----------|------|
| 問題解決型 | 技術記事、How-to | 問題提起 → 原因分析 → 解決策 → 実装 → 結果 |
| 比較型 | ツール比較、選定記事 | 背景 → 評価軸の定義 → 各選択肢の分析 → 結論 |
| 時系列型 | 体験記、ポストモーテム | 背景 → 経緯（時系列） → 学び → 今後 |
| 概念説明型 | 入門記事、解説 | なぜ重要か → 概念の定義 → 具体例 → 応用 |
| 主張型 | 意見記事、エッセイ | 主張の提示 → 根拠1,2,3 → 反論への応答 → まとめ |
| 発見型 | 調査記事、学習記録 | きっかけ → 調査過程 → 発見 → 考察 |

### Step 4: 構成案を出力する

以下のフォーマットで構成案を出力する。

---

## 構成案

**テーマ**: （一文で）
**想定読者**: （誰が、どんな状態で読むか）
**読後のゴール**: （読者がどうなっていてほしいか）
**構成パターン**: （選択したパターン名）
**想定文字数**: （目安）

### アウトライン

各セクションについて以下を示す。

```
## セクション見出し（仮）
- このセクションの役割: （読者の思考フローの中でこの部分が果たす役割）
- 含める内容:
  - ポイント1
  - ポイント2
  - ...
- 想定分量: （全体に対する割合の目安）
```

### 構成のポイント

- なぜこの順番にしたのかの簡潔な説明
- 特に注意すべき点（ここで読者が離脱しやすい、ここが記事の山場、など）

### 代替案（任意）

明確な別の構成が考えられる場合は、簡潔に代替案も示す。

---

## 心がけること

- **見出しは具体的に**: 「はじめに」「まとめ」のような汎用的な見出しではなく、その記事ならではの内容が伝わる見出しを提案する。「はじめに」の代わりに「なぜ今Rustのエラー処理を見直すのか」のように。
- **冒頭で読者を掴む**: 最初のセクションは記事の存在意義を示す場所。読者が「自分に関係ある」と感じられるフックを設計する。
- **1セクション1メッセージ**: 各セクションが伝えるメッセージは一つに絞る。複数のことを言いたい場合はセクションを分ける。
- **具体と抽象の往復**: 抽象的な説明の後には具体例、具体例の後には一般化。この往復が理解を深める。
- **適切な粒度**: 記事の長さやテーマに対してセクションが多すぎたり少なすぎたりしないよう注意する。
