# 開発ワークフローの自律オーケストレーション

## このスキルが存在する理由

ユーザーの CLAUDE.md は「claude はオーケストレータ、codex は調査・レビュー、cursor は実装」
という三者協力モデルを規定しており、**相互確認がコンセプトの核心**である。claude が自分で
手を動かすと相互検証が成立しない。

過去に発生した失敗:

- `main` ブランチ上で直接 CI ワークフローを編集した
- Issue を作らずに作業へ着手した
- 「軽い調査だから」と codex 委譲を省略し grep を繰り返した
- auto mode の「即時実行」を盾に計画レビューを飛ばした

これらは**一つ一つは部分的に正当な判断**だが、積み重ねるとワークフローの骨格が崩れる。
このスキルは、その骨格を明示的なチェックポイントで再構築する。

---

## 運用原則: 全自動・仮決め即実行

ユーザーは「原則、全部自動で進める」ことを期待している。**確認ゲートは置かない**。

- ブランチ名・Issue タイトル / 種別 / スコープ・起票粒度が曖昧でも、claude が
  仮決めして進める。仮決めの根拠を一文で宣言してから実行する
- 事前確認ではなく**事後報告**でユーザーに差し戻し余地を残す
- ワークフローの骨格 (Phase 1→6 の順序、codex/cursor 委譲、worktree) は崩さない。
  骨格をスキップするのは禁止。骨格「内部」の確認のみ省略する

### 確認が許される例外

以下の場合に限り、事前にユーザーへ確認する:

- 破壊的で取り返しがつかない操作 (force-push, 本番データ削除, 公開 API 破壊的変更)
- ユーザーがそのセッションで明示的に「確認してから進めて」と指示した場合
- 外部サービスへの課金や副作用を伴う操作

「曖昧だから一応確認しておこう」は **禁止**。仮決めして走る。

## Phase 1: リクエスト分類

ユーザーの直近メッセージを読み、分類する:

| 分類 | シグナル | 進むべき Phase |
|------|---------|---------------|
| 実装タスク | 「実装」「修正」「追加」「書いて」「作って」「リファクタ」「直して」 | 2 → 3 → 4 → 5 → 6 |
| 調査/計画タスク | 「調査」「調べて」「原因」「仕組み」「計画」「影響範囲」「設計検討」 | 2 → 3 のみ |
| 例外パス単体変更 | README 追記、`.github/workflows/*.yml` 修正、`.claude/**` 設定変更など、対象が単一の例外パスに閉じる | 2 → 4 (claude 直接) |
| 会話/質問 | 「どう思う？」「なぜ？」「〜について教えて」 | 通常応答。スキル終了 |

判定に迷ったら**より重い分類**を選ぶ。過剰な委譲のコストより、骨格崩壊のコストの方が大きい。

---

## Phase 2: 前提確認 (ブランチ + Issue)

Phase 3〜5 に進むすべての分類で必ず通過する。`main`/`master`/既存ブランチの状態を確認し、
Issue の有無も確認する。

### 2-1. ブランチ確認 (worktree 前提)

```bash
git branch --show-current
```

**重要: `enforce-worktree.sh` hook が `git checkout -b` / `git switch -c` /
`git branch <name>` をブロックする。** ブランチ作成は以下のいずれかを使う:

- `git worktree add -b <branch> <path> <start-point>`
- `EnterWorktree` ツール (ToolSearch で schema 取得後に呼び出し)

**`<path>` の選定は `superpowers:using-git-worktrees` スキルが正本** であり、
以下の優先順位で決める (親ディレクトリを汚すので `../<repo>-<branch>` は使わない):

1. 既存の `.worktrees/` ディレクトリがあればそこに作る (推奨の project-local hidden)
2. 既存の `worktrees/` があればそれを使う
3. CLAUDE.md に worktree ディレクトリ指定があればそれに従う
4. どれも無ければ `.worktrees/<branch>` を既定として作成する。初回は
   `.gitignore` への追加+コミットが必要 (superpowers スキルが自動でやる)

迷ったら `superpowers:using-git-worktrees` スキルを明示的に呼び出す。この
スキルは directory selection・`.gitignore` 安全確認・baseline test まで面倒を見る。

結果が `main` / `master` / `trunk` の場合:

1. タスクから仮のブランチ名を決める (例: `fix/ci-path-filter`, `feat/user-auth`,
   `refactor/font-loader`, `docs/readme-update`)。命名が一意に決まらなくても**仮決めで進む**
2. ブランチ名と一文の根拠を宣言する
3. そのまま worktree を作成し、作業ディレクトリとして使う
4. 確認は求めない

すでに作業ブランチ上にいる場合はそのまま進む。ブランチ名がタスクと明らかに乖離
している場合 (`fix/auth` 上で CI 修正など) は新 worktree を切るが、これも確認不要。
作業後の報告でユーザーが差し戻せばよい。

### 2-2. Issue 確認

```bash
gh issue list --search "<タスクのキーワード>" --state open --json number,title,state,url
```

#### 関連 Issue が見つかった場合

- タイトルと本文を `gh issue view` で確認
- 内容が今回のタスクと十分に合致していれば番号を記録し Phase 3 に進む
- 微妙にズレていても claude が仮判断で「紐付ける / 新規起票する」を選び即実行する。
  事後に選択理由を報告する

#### 関連 Issue が見つからなかった場合

**`/create-issue` に委譲する**。このスキルは種別仮分類・タイトル生成・本文生成・`## 明確にすべきポイント` チェックリスト付与・`gh issue create` 実行までを一気通貫で行う。

- 呼び出し時に「dev-orchestration の Phase 2-2 から」という文脈と、判明しているタスク内容を渡す
- 戻り値の Issue 番号を Phase 3 以降に引き継ぐ
- 起票内容は `/create-issue` が Phase 5 で事後報告済み。claude 側の重複報告は不要
- プレースホルダが多く残った場合は `/create-issue` 側で `/refine-issue` 提案を出す

**claude が自分で `gh issue create` を呼ばない**。タイトル命名規則・本文テンプレ適用・明確にすべきポイント集約のノウハウは `/create-issue` に集約されているため、重複実装を避ける。

### 2-3. 例外: 極めて軽微な変更

すべてを満たす場合のみ Issue 起票を省略してよい:

- 変更対象が単一ファイルかつ Phase 4 の例外パス表に含まれる
- 変更内容が typo 修正 / 定数値の軽微な調整 / 追記のみ
- 機能仕様・CI/デプロイ挙動・公開 API に影響しない

「軽微だと思い込みたい」という感覚自体がレッドフラグ。確信が持てないなら起票側に倒す。
ブランチ確認 (2-1) はこのケースでもスキップしない。

## Phase 3: 調査フェーズ (codex 委譲)

以下のいずれかに該当する場合、`/codex-investigate` に委譲する:

| トリガー | 例 |
|---------|-----|
| ファイル 3 つ以上を読む必要がある | 「認証フローを理解したい」 |
| 仕組み・設計の説明を求められた | 「なぜこのデータ構造なのか」 |
| 根本原因の特定 | 「このバグの原因を特定して」 |
| 影響範囲の分析 | 「この関数を変えたら何が壊れる？」 |
| 実装計画の策定 | 「〜を実装する計画を立てて」 |

claude が自分で grep / read を 3 回以上繰り返したくなったら、その時点で止めて委譲する。
claude が直接読んでよいのは**ブランチ・Issue 判定に必要な最小限** (例: `git status`,
`ls`, `CLAUDE.md` 冒頭部, 1〜2 ファイル程度の確認) のみ。

### 委譲スキップが許される稀なケース

- 直近の会話で同じファイルの内容を既に読んでおり追加調査が不要
- 単一ファイル内の既知の箇所への局所的な修正で、周辺コンテキストが自明

このスキップを行った場合、**なぜ委譲しなかったか**を一文でユーザーに報告する。

## Phase 4: 実装フェーズ

### 4-1. 例外パス判定 (claude 直接編集の可否)

変更対象が**すべて**以下に該当する場合のみ、claude が直接編集してよい:

| カテゴリ | パス/拡張子 |
|---------|------------|
| ドキュメント | `*.md`, `*.txt`, `*.rst`, `*.adoc` |
| 設定ファイル | `*.json`, `*.yaml`, `*.yml`, `*.toml`, `*.ini`, `*.env.example` |
| CI/CD | `.github/**` |
| Claude 自身の設定 | `.claude/**`, `~/.claude/**` |

**1 ファイルでもソースコード拡張子 (`*.rs`, `*.ts`, `*.py`, `*.go`, `*.java`,
`*.rb`, `*.c`, `*.cpp`, 等) が混ざる場合、例外は適用されず全体を `/cursor-code`
に委譲する。** これは `~/.claude/hooks/delegate-coding.sh` でも物理的にブロックされる。

### 4-2. 計画レビュー (codex, 実装前)

以下のいずれかに該当する場合、cursor 呼び出し**前**に計画レビューを受ける:

- 変更ファイルが 3 つ以上
- 設計判断 (新しい抽象, 依存関係の変更, インターフェース設計) を含む
- 破壊的変更 (公開 API, データ構造, 設定互換性の変更)
- セキュリティ・認証・課金など影響が大きい領域

実装計画を文章化し `/codex-investigate` に「以下の実装計画の妥当性、抜け漏れ、
リスクをレビューして」と依頼する。軽微な実装では省略してよいが、**省略した判断自体を
ユーザーに報告する** (「計画レビューを省略しました。理由: 単一ファイルへの局所修正のため」)。

### 4-3. cursor 委譲 (例外パス以外のすべての実装)

`/cursor-code` を呼び出す。その際:

- Phase 3 の調査結果を context として渡す
- 変更を許可するファイル/範囲を明示する
- テスト更新・追加の方針を明示する
- TDD が適用可能なタスクでは Red → Green → Refactoring の順で進めるよう指示

### 4-4. claude 直接編集 (例外パスのみ)

4-1 で例外パス判定に該当した場合、claude が Edit/Write してよい。ただし:

- 変更は最小限に留める (関係ない箇所のついでの修正は避ける)
- 変更理由を commit 時に英語で (公開リポジトリの場合) 明記できるよう整理する

## Phase 5: 実装後レビュー + 動作検証（自律実行）

cursor が成果物を返したら:

1. `git diff` / `git status` で変更内容を確認する
2. 以下のいずれかに該当するなら **`codex:rescue` スキル** (差分レビュー) を呼ぶ:
   - 変更行数が多い (目安: 合計 100 行超)
   - 設計判断を含む
   - 意図と差分にズレを感じる
3. 軽微な差分は claude が目視確認し、気になる点があればユーザーに報告

**重要: 差分レビュー目的で `codex exec` を Bash 直叩きしない。** `codex:rescue` スキルが runtime hook で stdin / 無音 / timeout を見張ってくれる。直接 `codex exec ... | tail` を起動すると stdin EOF を待って永久 hang した実績がある。`codex:rescue` を経由すれば runtime が stdin 閉じ・ファイル出力・進捗監視を一括で担保する。

### 5-1. 動作検証は claude が自律実行する

**ユーザーに「あなた側で動かして確認してください」と委ねない**。変更に対応する動作確認は claude が実機で実行する:

| 変更種別 | 自律検証アクション |
|---------|------------------|
| コード変更 | 該当機能を実行 / テスト（`cargo test`, `npm test`, 実行バイナリ起動）し、期待出力を得るまで回す |
| 手順ドキュメント変更 | 文書どおりにコマンドを打って再現する（例: README の Getting Started、notes の手順） |
| API / サーバー変更 | 起動 → リクエスト → レスポンス検証までを 1 セット（`wrangler dev`, `cargo run`, `docker compose up` など） |
| ビルド / CI 変更 | 変更後のフルビルドを実行（`cargo check --all-targets`, `npm run build`） |

動作検証が claude の環境で **実行不可能** な場合のみ、その理由を明示してユーザーへ依頼する:

- 外部有料サービスへのアクセスが必要（本番 DB, 課金 API）
- ユーザー個人の認証情報が必要（特定の SSO, 社内ネットワーク内リソース）
- 本番環境でのみ再現するデータ依存

「時間がかかるから」「ビルドが重いから」「環境依存が怖いから」は **実行不可能の理由にならない**。バックグラウンド実行（`run_in_background`）や poll loop で待機すれば済む。

検証結果は Phase 5 の判断材料 / Phase 6 報告の動作確認エビデンスに組み込む。

## Phase 6: 統合と報告（コミット + push + PR まで自律完走）

Phase 5 が merge 可の判断に至ったら、**ユーザーの明示指示を待たずに以下まで連続で仮決め実行する**:

1. `/git-commit` に委譲（差分グルーピング + メッセージ生成）
2. push（ブランチ未トラッキングなら `git push -u origin <branch>`）
3. `/create-pr` に委譲（PR 起票）

ブランチ名・コミット粒度・コミットメッセージ・PR 本文は既存慣習から自動生成。事後報告でユーザーに
差し戻し余地を残す。`feedback_full_automation` が定める「仮決め即実行」は Phase 6 にも適用される。

作業完了時の報告内容:

- 作業ブランチ名
- 参照 / 起票した Issue 番号
- 委譲先（codex / cursor）と各々の処理サマリー
- 直接編集した場合はそのパスと理由
- 作成した PR URL
- 残課題があれば次のアクション候補

### 例外: 事前確認が必要な Phase 6 操作

以下の**破壊的 / 取り返しがつかない**操作のみ事前確認する。それ以外は確認ゲート禁止:

- `main` / `master` / 保護ブランチへの force-push
- 既存の published タグ / リリースの書き換え
- 他コラボレーターのブランチへの push
- 公開済み PR の close（ユーザー同意なし）

### 6-1. `/git-commit` 委譲（Phase 6 内で常時実行）

commit の粒度判断・メッセージ生成・既存慣習への適合は `/git-commit` に集約。claude は
`git add` / `git commit` を直接呼ばない。

### 6-2. `/create-pr` 委譲（Phase 6 内で常時実行）

push 未実施なら `/create-pr` 側で無条件 push。既存 PR 命名規則・テンプレートを学習し、
`## Summary` / `## 動作確認` / `## Test Plan` / `## レビュー時の重点確認ポイント` /
`Closes #N` を含む本文を生成して `gh pr create` を実行する。draft/ready は 500 行 +
設計判断有無で仮決め。claude は `gh pr create` を直接呼ばない。

Phase 2-2 で起票 / 参照した Issue 番号と、Phase 5 で codex レビューが見つけた論点（あれば）
を `/create-pr` 呼び出し時の文脈として渡すと、動作確認 / レビュー重点ポイント生成の質が上がる。

**プロジェクト固有の PR ルール**: プロジェクトの `CLAUDE.md` や `.claude/` 配下に
「PR 作成時の必須ルール」「PR 本文チェックリスト」等が書かれている場合、`/create-pr` を
呼ぶ**前に**チェックリスト項目を全満たしていることを確認する。未達なら先にそれを解消してから
Phase 6-2 に戻る（例: mojiemoji では `## サンプル` セクション + 実生成 GIF/PNG の埋め込み必須）。

### /loop との関係

`/loop` で dev-orchestration を繰り返すケースでも動作は同じ（1 イテレーションが PR URL
報告まで完走する）。従来あった「/loop では例外的に完走する」という言い回しは不要。

## レッドフラグ集

以下の思考が浮かんだら、それは**このスキルが必要だった兆候**。浮かんだ時点で
Phase 1 から再起動する:

| 内部思考 | 実態 |
|---------|------|
| 「ブランチ名を確認しておこう」 | ユーザーは「仮決めで進めて」と明言済。確認禁止。命名して走る |
| 「Issue の粒度が迷うから先に聞こう」 | 仮決めで起票し、事後報告で差し戻し可能にする |
| 「.github/** は例外だから Issue もブランチも不要」 | 例外は**編集委譲**の話のみ。ブランチ・Issue 判定は別レイヤー |
| 「自分で grep した方が早い」 | 3 ファイル以上なら codex。速度より相互検証の価値が高い |
| 「計画レビューは今回は省略でいいだろう」 | 省略自体を記録してユーザーに見せる |
| 「skill を呼ぶほどではない」 | 呼ばずに失敗した実績があるからこのスキルが存在する |
| 「軽微な変更だから手順は不要」 | 「軽微」の判定も Phase 2〜4 の手順で行うもの |
| 「git checkout -b で十分だろう」 | `enforce-worktree.sh` でブロックされる。最初から `git worktree add -b` を使う |
| 「ユーザーが急いでいそうだから省略」 | 急ぐときこそ手戻りのコストが大きい (ただし事前確認のための停止は**禁止**) |
| 「コミット前にユーザー確認を取ろう」 | Phase 6 は自律完走が原則。破壊的操作以外は確認禁止 |
| 「PR 起票は別ターンでいいだろう」 | Phase 6 に PR 起票まで含まれる。1 ターンで完走する |
| 「動作確認はユーザーに任せよう」 | Phase 5-1 で claude が自律実行する。`run_in_background` と poll で待機すれば時間の問題は解決 |
