# 新規Issue起票

## ユーザー入力

```text
$ARGUMENTS
```

`$ARGUMENTS`が空の場合は直近の会話コンテキストからタスク内容を推論する。

## 目的

**「このIssueだけ読んで着手者が最低限の意図を把握でき、かつ何が未確定かが一目で分かる」**状態で新規Issueを起票する。
完璧を目指さず、仮決めで即起票する。未確定項目は本文末尾の **「明確にすべきポイント」チェックリスト** として残し、`/refine-issue` に具体的なハンドオフを渡せる形にする。

このスキルが存在する理由:

- dev-orchestrationのPhase 2-2から呼ばれる「骨格チェックポイント」としての役割
- 「Issue作って」のような独立トリガーからも同じ品質で起票できる再利用性
- タイトル命名規則・本文テンプレ適用・種別仮分類のノウハウを一箇所に集約

---

## 運用原則: 仮決め即実行

- **確認ゲートは置かない**。タイトル・本文・種別が曖昧でも仮決めして起票する
- **事前確認ではなく事後報告**で、ユーザーが差し戻せる余地を残す
- 例外: 破壊的なラベル（`security`, `critical`等）を付けるかの判断、既存重複Issueの上書き相当の場合のみ確認

---

## Phase 1: 既存パターンの学習

起票前に、リポジトリの既存慣習を最小限読み取る。**ここで深く調査しない**。深い調査が必要なら`/codex-investigate`を先に呼ぶのが正しい（このスキル単体では扱わない）。

### 1-1. 既存Issue命名規則

```bash
gh issue list --state all --limit 20 --json number,title,labels
```

以下を読み取る:

- タイトルのprefix慣習（`feat:`, `fix:`, `[feat]`, `Add:`, prefix無し等）
- 言語（英語・日本語・混在）
- scope表記の有無（`feat(auth):` のような形式があるか）
- 多用されるラベル

**明確なパターンがある場合**はそれに従う。**混在・不統一**なら直近10件の多数派に合わせる。**履歴が乏しい**場合はConventional Commits形式（`<type>(<scope>): <description>`）をフォールバックとする。

### 1-2. Issueテンプレートの確認

```bash
ls .github/ISSUE_TEMPLATE/ 2>/dev/null
```

- テンプレートが存在する場合、タスク種別に対応するテンプレート（`feat.md`, `bug_report.yml`等）を読み込み、セクション構造を採用する
- テンプレートが無い場合、Phase 3のフォールバック構造を使う

### 1-3. 重複チェック

タスクのキーワードで既存Open Issueを検索する:

```bash
gh issue list --search "<キーワード>" --state open --json number,title,url
```

**類似Issueが見つかった場合:**

- タイトルと概要を一文で要約してユーザーに報告する
- 「このIssueに追記しますか、それとも別Issueとして起票しますか」と**短く確認**する（ここは仮決めしない。重複起票はノイズになるため）
- 追記方針なら`/refine-issue`に委譲、別起票ならPhase 2以降に進む

---

## Phase 2: 種別仮分類

タスク内容から種別を仮決めする。`refine-issue`と同じテーブルを使う:

| 種別 | トリガー語 | カテゴリ |
|------|-----------|---------|
| `feat` | 追加、新機能、実装、新しい | 機能系 |
| `refactor` | リファクタ、再設計、整理 | 機能系 |
| `perf` | 高速化、パフォーマンス、最適化 | 機能系 |
| `fix` | バグ、修正、直す、動かない | 不具合系 |
| `hotfix` | 緊急、本番障害、critical | 不具合系 |
| `docs` | ドキュメント、README、説明 | 運用系 |
| `chore` | 設定、依存、雑務、運用 | 運用系 |
| `ci` | CI、GitHub Actions、ワークフロー | 運用系 |
| `test` | テスト追加、カバレッジ | 運用系 |

判定順序:

1. ユーザー発話の明示的なキーワード
2. 影響を受けるファイルパス（`.github/workflows/` → ci, `*.md` → docs 等）
3. タスク文脈からの推測

**判定に迷ったら`chore`で起票**して後で refine/edit すればよい。完璧な分類より起票のスピードを優先する。

## Phase 3: タイトルと本文の生成

### 3-1. タイトル生成

Phase 1で学習した命名規則に従う。フォールバックフォーマット:

```
<type>(<scope>): <簡潔な説明>
```

ルール:

- **簡潔に**（60文字以内を目安）
- **動詞で始める**（命令形: add, fix, update, remove）
- **「何を」より「なぜ/どう」が伝わる**ように
- **末尾にピリオドを付けない**

### 3-2. 本文生成

**テンプレートがある場合**: そのセクション構造を採用し、分かる範囲で埋める。不明セクションは`（要追記）`のプレースホルダを置き、かつ後述の「明確にすべきポイント」セクションを末尾に追加する。

**テンプレートが無い場合**: 以下のカテゴリ別フォールバック構造を使う。どの構造を選んでも、**末尾に「明確にすべきポイント」セクション（チェックリスト形式）を必須で付ける**。

#### 機能系（feat / refactor / perf）

```markdown
## 背景・動機

<なぜこの変更が必要か。現状の課題と期待する状態>

## ゴール

<改善後にどういう状態になれば完了か>

## スコープ

**やること:**
- 項目1
- 項目2

**やらないこと（非スコープ）:**
- 項目

## 補足

<関連リンク・参考情報。省略可>

## 明確にすべきポイント

<refine で詰めるべきオープン論点。チェックリスト形式。Phase 3-4 のルール参照>

- [ ] <論点1>
- [ ] <論点2>
```

#### 不具合系（fix / hotfix）

```markdown
## 現象

<何が起きているか。期待動作と実際の動作>

## 再現手順

1. ステップ1
2. ステップ2
3. ステップ3

## 環境

- OS / ブラウザ / バージョン等（分かる範囲）

## 原因の仮説

<分かる範囲で。不明なら「未調査」と明記>

## 影響範囲

<影響を受けるユーザー・機能・モジュール>

## 明確にすべきポイント

<refine で詰めるべきオープン論点。チェックリスト形式。Phase 3-4 のルール参照>

- [ ] <論点1>
- [ ] <論点2>
```

#### 運用系（docs / chore / ci / test）

```markdown
## 目的

<なぜ今この作業が必要か>

## 対象

<影響するファイル・ディレクトリ・ワークフロー>

## タスク

- [ ] 作業項目1
- [ ] 作業項目2

## 検証方法

<作業完了の確認方法>

## 明確にすべきポイント

<refine で詰めるべきオープン論点。チェックリスト形式。Phase 3-4 のルール参照>

- [ ] <論点1>
- [ ] <論点2>
```

### 3-3. 仮決めの原則

- 情報が足りないセクションは**空欄にせず**「未調査」「要追記」「要確認」等のプレースホルダを明記する
- プレースホルダが3つ以上残る場合、起票後に`/refine-issue`を提案する旨をPhase 5に引き継ぐ
- 対話は**最大1ラウンド、最大2項目**まで。それ以上聞きたくなったら「仮決めして起票 → refine で詰める」に切り替える

### 3-4. 「明確にすべきポイント」セクション（必須）

本文末尾に **`## 明確にすべきポイント`** を必ず付ける。これは **refine への明示的なハンドオフ** であり、「本文を厚く書かずに refine に渡す」設計を成立させる要。

**書き方のルール:**

- **チェックリスト形式**で書く（`- [ ] <論点>`）。refine 側で詰め終わった項目が可視化できる
- **項目は 3〜7 件**を目安にする（0 件ならそもそも仮決めしすぎ、10 件超なら粒度が細かすぎ）
- **一項目は一文で、具体的に**書く。抽象的な「仕様を詰める」は禁止、「redis のキャッシュ TTL を何秒にするか」のように決定可能な問いにする
- **「本文中のプレースホルダ」と重複してもよい**。チェックリストは「refine 側の質問候補」として機能するので、再掲で可視性が上がる
- 本文中で未調査にしていない項目でも、意思決定が必要な論点（選択肢が複数ある設計判断など）があれば含める

**カテゴリ別の典型パターン:**

| カテゴリ | よく出る論点 |
|---------|-------------|
| 機能系 (feat) | ユースケース優先順位、I/F 設計、既存実装との棲み分け、エラーハンドリング方針、テスト戦略 |
| 機能系 (refactor) | 移行期の互換性維持、段階的実施の分割粒度、既存利用箇所の扱い |
| 機能系 (perf) | 目標メトリクスの具体値、計測条件、受け入れ基準となる閾値 |
| 不具合系 (fix) | 再現環境、再現率、原因仮説の検証方法、修正スコープ（対症療法 vs 根本対処）、回帰テスト範囲 |
| 不具合系 (hotfix) | ワークアラウンドの有無、ロールバック手順、ユーザー影響 |
| 運用系 (docs/chore/ci/test) | 対象ファイルの確定、完了確認方法、ロールバック手順 |

**良い例:**

```
## 明確にすべきポイント

- [ ] password が空以外に `" "` (空白のみ) の場合も 400 を返すか（業務ルール要確認）
- [ ] 発生環境（本番 / ステージング / ローカル）と、該当デプロイ時期
- [ ] スタックトレースの添付（サーバーログからの抽出可否）
- [ ] 修正スコープを `/login` のみに閉じるか、他の認証エンドポイントも同時に直すか
```

**避ける書き方:**

```
## 明確にすべきポイント

- [ ] 仕様を詰める  ← 抽象すぎる
- [ ] 要確認        ← 何をか分からない
- [ ] テスト観点    ← 具体的な問いがない
```

## Phase 4: 起票

### 4-1. ラベル判定

Phase 2の種別と、既存ラベル一覧（`gh label list --json name,color`）から付与するラベルを決める。**存在しないラベルは作らない**（ラベル設計は別タスク）。

破壊的・政治的判断が必要なラベル（`security`, `breaking`, `critical`, `regression` 等）はユーザーに確認する。

### 4-2. 起票実行

本文はシェルエスケープ問題を避けるため一時ファイル経由で渡す:

```bash
cat > /tmp/issue-body-$$.md << 'ISSUE_EOF'
<生成した本文>
ISSUE_EOF

gh issue create \
  --title "<生成したタイトル>" \
  --body-file /tmp/issue-body-$$.md \
  --label "<ラベル1>,<ラベル2>"

rm -f /tmp/issue-body-$$.md
```

`gh issue create`はデフォルトでPR URLを返すので、そのURLとIssue番号を記録する。

### 4-3. エラーハンドリング

- 権限エラー → ユーザーに報告（認証状態の確認を促す）
- ラベル不存在エラー → 該当ラベルを除外してリトライ
- その他 → エラーメッセージをそのままユーザーに伝える

---

## Phase 5: 報告

ユーザーに以下を簡潔に報告する:

```
Issue #<番号> を起票しました: <URL>
タイトル: <生成したタイトル>
種別: <feat/fix/...>
ラベル: <付与したラベル>

明確にすべきポイント (N 項目):
- <本文末尾のチェックリストから主要 3 件を抜粋>
```

`## 明確にすべきポイント` の項目が 2 件以上ある場合は、続けて以下を提案する（強制はしない）:

```
本文の精緻化は `/refine-issue #<番号>` で行えます。
```

dev-orchestrationから呼ばれた場合は、Issue番号を呼び出し元に戻すだけで十分（ユーザー報告は呼び出し元が行う）。

## 例

### 例1: 明確な実装依頼

**ユーザー発話:**
> ログイン画面で空文字のパスワードを入力すると500エラーになるのを直したい

**生成:**

- 種別: `fix`
- タイトル: `fix(auth): return 400 instead of 500 for empty password`
- 本文:

```markdown
## 現象

ログイン画面で password フィールドを空文字のまま送信すると 500 Internal Server Error が返る（期待: 400 Bad Request）。

## 再現手順

1. `/login` にアクセス
2. email に任意の値を入力
3. password を空のまま Submit
4. 500 エラーレスポンス

## 環境

未調査（要追記）

## 原因の仮説

未調査。バリデーション前に空文字をパース処理に渡している可能性。

## 影響範囲

未調査（要追記）

## 明確にすべきポイント

- [ ] 空白のみ (`" "`) も 400 として扱うか（業務ルール確認）
- [ ] 発生環境（本番 / ステージング / ローカル）とデプロイ時期
- [ ] サーバーログからのスタックトレース添付可否
- [ ] 修正スコープを `/login` に閉じるか、他認証エンドポイントも同時対応か
- [ ] 回帰テスト範囲（既存のログインテストを拡張するか、新規テスト追加か）
```

起票後: `/refine-issue` 提案（明確にすべきポイントが 5 項目あるため）

### 例2: dev-orchestration経由の仮決め起票

**呼び出し元の文脈:**
> （dev-orchestration Phase 2-2）CI ワークフローの pull_request トリガーで path filter が効いていないので修正したい

**生成:**

- 種別: `ci`
- タイトル: `ci: fix path filter on pull_request trigger`
- 本文（運用系フォールバック、要追記多め）:

```markdown
## 目的

CI ワークフローの pull_request トリガーで path filter が効かず、意図しないジョブが実行される状態を解消する。

## 対象

- `.github/workflows/ci.yml`（要確認）

## タスク

- [ ] 現状の path filter 設定を確認
- [ ] 正しい filter に修正
- [ ] 動作確認

## 検証方法

未調査（要追記）

## 明確にすべきポイント

- [ ] 意図しないジョブの具体例（どのジョブがどの変更で走った）
- [ ] path filter に入れるべきパス一覧（`src/**`, `tests/**` など）
- [ ] フィルタを適用する対象イベント（`pull_request` だけか `push` も含むか）
- [ ] リグレッション確認方法（PR ドラフトで fire & check）
```

起票後、dev-orchestration に Issue 番号を返し、Phase 3 以降に引き継ぐ。

## 他スキルとの関係

| 相棒 | 役割 | このスキルとの関係 |
|------|------|-------------------|
| `/refine-issue` | 既存Issueの精緻化 | 起票後にプレースホルダ多めなら提案する |
| `dev-orchestration` | ワークフロー判断ハブ | Phase 2-2 からこのスキルを呼ぶ |
| `/codex-investigate` | コードベース調査 | 事前調査が必要なら先に呼ばれる。このスキルは調査しない |
| `/create-pr` | PR作成 | 対になる入口。Issue と PR で責務分担 |

## レッドフラグ

以下が頭をよぎったら、仮決めで起票して refine に回す:

| 思考 | 実態 |
|------|------|
| 「もっと詳しく聞いてから起票しよう」 | 対話は最大1ラウンド。refineで詰める |
| 「完璧な本文が書けないから起票しない」 | プレースホルダを置いて起票する |
| 「類似Issueがあるか念のため全部読もう」 | 検索コマンド1発の結果で判断する |
| 「ラベルを新設すべきかも」 | 既存ラベルで起票。ラベル設計は別タスク |
