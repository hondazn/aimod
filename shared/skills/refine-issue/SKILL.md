# Issue精緻化

## ユーザー入力

```text
$ARGUMENTS
```

作業を開始する前に、ユーザーからの入力を理解する。

## 目的

既存のGitHub Issueを、「このIssueだけ読めば実装に着手できる」状態にする。
プロジェクトのIssueテンプレートとの照合・種別判定・コードベース調査を踏まえて不足情報を特定し、ユーザーとの対話を通じて仕様を精緻化してIssueを更新する。

---

## Phase 1: Issue取得と分析

### 1-1. Issue番号の抽出

`$ARGUMENTS` からIssue番号を抽出する。以下の形式に対応:
- `#123`
- `123`
- `https://github.com/.../issues/123` 形式のURL

番号が特定できない場合はユーザーに確認する。

### 1-2. Issue情報の取得

```bash
gh issue view <番号> --json title,body,labels,assignees,milestone,state
```

**エラーハンドリング:**
- Issueが存在しない場合 → ユーザーに番号の確認を求める
- CLOSEDの場合 → 「このIssueはクローズ済みです。精緻化を続けますか？」と確認する
- 権限エラーの場合 → リポジトリとの認証状態を報告する

### 1-3. 種別判定

タイトルのConventional Commitプレフィックスとラベルから種別を判定する:

| プレフィックス | ラベル例 | 種別 | カテゴリ |
|--------------|---------|------|---------|
| `feat:` | `enhancement` | feat | 機能系 |
| `refactor:` | `refactoring` | refactor | 機能系 |
| `perf:` | `performance` | perf | 機能系 |
| `fix:` | `bug` | fix | 不具合系 |
| `hotfix:` | `critical`, `hotfix` | hotfix | 不具合系 |
| `docs:` | `documentation` | docs | 運用系 |
| `chore:` | `chore` | chore | 運用系 |
| `ci:` | `ci` | ci | 運用系 |
| `test:` | `testing` | test | 運用系 |
| `release:` | `release` | release | 運用系 |

判定順序:
1. タイトルの `種別:` プレフィックスを最優先
2. ラベルで補完
3. 上記で判定不能な場合、Issue本文のキーワード（「バグ」「不具合」→fix、「追加」「新機能」→feat 等）から推測し、推測結果をユーザーに確認する

### 1-4. テンプレート照合

`.github/ISSUE_TEMPLATE/` をファイルパターン検索でスキャンし、種別名に対応するテンプレートを検索する。

- 種別名がファイル名に含まれるテンプレートを読み込む（例: `feat.md`, `fix.yml`）
- YAML形式（`*.yml`）の場合はフィールド定義からセクション構造を把握する
- テンプレートが見つからない場合は以下の汎用セクションで分析する:

| セクション | 確認内容 |
|-----------|----------|
| 概要 | 何を実現/修正するかの説明があるか |
| 背景・動機 | なぜこの変更が必要かの説明があるか |
| 完了条件 | 具体的・検証可能な完了基準があるか |
| スコープ | やること/やらないことが明確か |
| 技術的詳細 | 実装に必要な情報（API仕様、データ構造等）があるか |

### 1-5. 充足度評価

テンプレートのセクション構造（またはフォールバックセクション）とIssue本文を照合し、各セクションの充足度を4段階で評価する:

| 充足度 | 記号 | 説明 |
|--------|------|------|
| 空欄 | `[ ]` | セクションが空、またはプレースホルダのまま |
| テンプレ残り | `[~]` | テンプレートの説明文がそのまま残っている |
| 薄い | `[△]` | 記述はあるが具体性が不足 |
| 充足 | `[x]` | 十分な情報がある |

評価結果を以下の形式でユーザーに報告する:

```
Issue #123: feat: ユーザー認証機能の追加
カテゴリ: 機能系（feat）

充足度:
[x] 概要 — 十分な記述あり
[△] 完了条件 — 記述はあるが検証可能な形式でない
[ ] スコープ — 記述なし
[~] 技術的詳細 — テンプレートの説明文のまま
```

## Phase 2: コードベース調査

Issueの文脈を理解するために、関連コードを調査する。この情報はPhase 3で具体的な選択肢を提示するために使う。

### 2-1. キーワード検索

Issueのタイトル・本文からキーワード（リソース名、機能名、モジュール名等）を抽出し、コード検索・ファイルパターン検索で関連コードを検索する。

検索対象:
- キーワードに一致するファイル名・ディレクトリ名
- キーワードに一致するコード内の定義（関数、型、モジュール等）
- 類似機能の実装パターン（ディレクトリ構造、命名規則）

不具合系（fix/hotfix）の場合は最近の変更履歴も確認する:
```bash
git log --oneline -20 -- <関連パス>
```

### 2-2. プロジェクト規約の確認

以下のファイルが存在する場合、**関連する箇所のみ**を確認する（ファイル全体を読む必要はない）:

- `CONTRIBUTING.md` — Issue/PR記述の規約
- AIアシスタント向け設定ファイル（`CLAUDE.md`, `.cursorrules` 等） — 開発ガイドライン
- `docs/` 配下の設計ドキュメント（Issueに直接関連するもののみ）

### 2-3. 調査結果の活用方針

調査結果はPhase 3で以下のように活用する:
- 「既存の `src/auth/service.ts` と同様のパターンにしますか？」のように具体的なコードを引用して選択肢を提示
- 「関連する実装は `src/xxx/yyy.rs` にあります」のように所在を明示
- 「既存の命名規則は `camelCase` です」のように規約を補足

## Phase 3: 対話的補完

Phase 1-2の結果から不足情報を特定し、ユーザーに質問する。
以下のカテゴリ別チェックリストで不足項目を洗い出し、質問優先順位に従って質問を組み立てる。

### 共通チェック項目

全種別で確認する基本項目。これが欠けていると実装に着手できない。

| 項目 | 確認内容 |
|------|----------|
| スコープ | やること/やらないことが明確か。Issue粒度は適切か（1PRで完結する大きさか）。依存Issueはあるか |
| 完了条件 | 具体的・検証可能・チェックリスト形式で記述されているか。「何をもって完了とするか」が第三者にも判断できるか |

### 機能系チェックリスト（feat / refactor / perf）

「何を改善するか」が中心。ゴール・アプローチ・完了条件が重要。

**ユースケース / 課題:**
- [ ] **誰が**このIssueの恩恵を受けるか（ユーザー種別・ペルソナ）
- [ ] **何を**実現/改善したいか（現状の課題 → 期待する状態）
- [ ] **なぜ**今この変更が必要か（ビジネス背景・技術的動機）

**ゴール:**
- [ ] 改善後の期待状態が具体的に記述されているか
- [ ] **perf** の場合: 定量的な目標値があるか（レイテンシ、スループット等）
- [ ] **refactor** の場合: リファクタリング後の設計目標が明確か

**アプローチ:**
- [ ] 実装方針・設計アプローチの概要があるか
- [ ] 大きな変更の場合: 段階的な実装計画があるか
- [ ] 既存コードへの影響範囲が把握されているか

**テスト観点:**
- [ ] 正常系テストケースの概要
- [ ] 異常系テストケース（バリデーションエラー、権限エラー等）
- [ ] エッジケース（空入力、境界値、大量データ等）

### 不具合系チェックリスト（fix / hotfix）

「何が起きているか」が中心。再現・原因・影響範囲が重要。

**再現性:**
- [ ] 再現手順が具体的に記述されているか（ステップバイステップ）
- [ ] 再現環境（OS、ブラウザ、バージョン等）が記載されているか
- [ ] 期待される動作 vs 実際の動作が明確か
- [ ] 再現率はどの程度か（常時/間欠的/特定条件下）

**原因分析:**
- [ ] 原因の仮説があるか
- [ ] エラーログ・スタックトレースが添付されているか
- [ ] 原因と疑われる最近の変更があるか（git log調査結果を活用）
- [ ] 関連するコードパスが特定されているか

**影響範囲:**
- [ ] 影響を受けるユーザー/環境の範囲
- [ ] 影響を受けるモジュール/機能の範囲
- [ ] 修正による副作用リスクの評価
- [ ] **hotfix** の場合: ワークアラウンドの有無

**テスト観点:**
- [ ] バグを再現するテストケース
- [ ] 修正後の期待動作を検証するテストケース
- [ ] 回帰テスト方針（関連機能への影響確認）

### 運用系チェックリスト（docs / chore / ci / test / release）

「何をどう作業するか」が中心。対象・タスク一覧・検証方法が重要。

**対象特定:**
- [ ] 影響するファイル/ディレクトリが特定されているか
- [ ] **ci** の場合: 影響するワークフロー/パイプラインが明確か
- [ ] **release** の場合: 対象バージョンとリリース内容が明確か
- [ ] **docs** の場合: 対象ドキュメントと更新箇所が特定されているか

**目的・必要性:**
- [ ] なぜ今この作業が必要か（きっかけ・背景）
- [ ] この作業を行わない場合のリスク/影響は何か

**タスク一覧:**
- [ ] 具体的な作業手順がリスト化されているか
- [ ] 作業の実施順序に依存関係がある場合、順序が明記されているか
- [ ] **release** の場合: リリース手順・チェックリストがあるか

**検証・ロールバック:**
- [ ] 作業完了の確認方法が明確か
- [ ] **ci** の場合: パイプラインの動作確認方法があるか
- [ ] 問題発生時の切り戻し手順が必要か（必要な場合、記載されているか）

### カテゴリ別の質問優先順位

不足項目の中から、以下の優先順位で質問を選ぶ。上位の項目ほど実装への影響が大きい。

| 機能系 | 不具合系 | 運用系 |
|--------|---------|--------|
| 1. 完了条件 | 1. 再現性 | 1. 対象特定 |
| 2. ユースケース/ゴール | 2. 原因分析 | 2. タスク一覧 |
| 3. スコープ | 3. 影響範囲 | 3. 検証方法 |
| 4. アプローチ | 4. スコープ | 4. 目的・必要性 |
| 5. テスト観点 | 5. テスト観点 | 5. ロールバック |

### 質問のルール

- **1回の質問で最大3問**（ユーザーへの質問機能を使用）
- **最大3ラウンド**で収束させる（完璧を目指さない。実装に着手できるレベルで十分）
- 充足度が「充足」の項目はスキップする
- コードベース調査結果を踏まえた具体的な選択肢を提示する（抽象的な質問は避ける）
- ユーザーの回答が曖昧な場合は、こちらから仮説を立てて確認する形にする

### 質問の組み立て方

1. 上記の質問優先順位に従い、不足項目を優先度順に並べる
2. 上位3項目を1回の質問にまとめる
3. 各項目について、コードベース調査結果を踏まえた具体的な選択肢や仮説を提示する

**質問テンプレート:**
```
Issueの精緻化にあたり、以下を確認させてください。

1. [項目名]: [コードベースの調査結果を踏まえた具体的な質問/仮説]
2. [項目名]: [同上]
3. [項目名]: [同上]
```

### 質問の具体例

**機能系:**
> 完了条件について確認させてください。以下のような理解で合っていますか？
>
> 1. `UserService.create()` に email バリデーションを追加する
> 2. 重複emailの場合は `409 Conflict` を返す
> 3. 既存の `src/validators/email.ts` のバリデーションロジックを再利用する
>
> 他に追加すべき条件があれば教えてください。

**不具合系:**
> 再現手順を整理したいのですが、以下の理解で合っていますか？
>
> 1. `/api/users` に POST リクエストを送信
> 2. `name` フィールドに空文字列を設定
> 3. 500 Internal Server Error が返る（期待: 400 Bad Request）
>
> `git log` を確認したところ、3日前の `abc1234` で `UserController` のバリデーション処理が変更されています。これが原因の可能性はありますか？

**運用系:**
> CIワークフローの変更対象を確認させてください。
> `.github/workflows/` に以下の3ファイルがあります:
> - `ci.yml` (テスト実行)
> - `deploy.yml` (デプロイ)
> - `lint.yml` (リンター)
>
> 今回変更するのは `ci.yml` のみという理解で合っていますか？

**避けるべき質問:**
- 「完了条件を教えてください」（漠然としすぎ）
- 「スコープはどうしますか？」（何について答えればよいか不明）
- 「テスト観点を挙げてください」（ユーザーに丸投げ）

## Phase 4: Issue更新

### 4-1. 更新内容のプレビュー

更新後のIssue body全文をマークダウンでプレビュー提示する。
差分がわかるように、追加・変更した箇所を明示する。

### 4-2. フォーマット方針

**共通ルール:**
- テンプレートのセクション見出し（`##`）は維持する
- 既存記述は保持し、追記・拡充する形で更新する
- 完了条件はチェックリスト `- [ ]` 形式で具体的な動作を記述する
- テンプレートにない場合は適切な見出しを追加する

**カテゴリ別の重点:**

| カテゴリ | 重点的に追記するセクション |
|---------|--------------------------|
| 機能系 | 完了条件（動作チェックリスト）、実装メモ/技術的詳細、スコープ外 |
| 不具合系 | 再現手順（ステップバイステップ）、期待動作 vs 実際の動作、原因分析、影響範囲 |
| 運用系 | タスク一覧（チェックリスト）、検証方法（コマンド例）、ロールバック手順 |

### 4-3. ラベルの整備

種別判定の結果に基づき、不足しているラベルがあれば追加を提案する:
```bash
gh label list --json name
```
で既存ラベルを確認し、適切なラベルがあれば追加を提案する。

### 4-4. ユーザー承認と更新

1. 更新内容の承認をユーザーに求める
2. 承認後、一時ファイル経由でIssueを更新する:

```bash
# 一時ファイルに本文を書き出してから更新する（シェルエスケープ問題を回避）
cat > /tmp/issue-body-<番号>.md << 'ISSUE_EOF'
(更新後のIssue body)
ISSUE_EOF
gh issue edit <番号> --body-file /tmp/issue-body-<番号>.md
rm /tmp/issue-body-<番号>.md
```

この方法を使う理由: `--body` に直接文字列を渡すと、マークダウンの特殊文字（バッククォート、ドル記号等）がシェルに解釈されて本文が壊れるため。

3. ラベル追加が承認された場合:
```bash
gh issue edit <番号> --add-label "label1,label2"
```

4. 変更箇所のサマリーを報告する:
   - 追記・修正したセクション
   - 主な追加情報
   - 追加したラベル（あれば）
   - 残課題（あれば）
