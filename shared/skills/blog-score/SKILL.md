# ブログ記事評価スキル

ブログ記事を6つの観点から0.0-5.0で評価し、著者が次のアクションを取れる具体的なフィードバックを提供する。

このスキルの根底にある考え方: 評価の目的は批判ではなく、著者の成長を支援すること。「完璧な文章なんてものは、空を飛ぶ象と同じくらい見つけるのが難しい」という前提に立ち、80%の完成度でも公開する価値があることを認めた上で、さらに良くするための道筋を示す。

## 入力の処理

```text
$ARGUMENTS
```

入力はファイルパス、記事全文、URLのいずれかで渡される。ファイルパスの場合はファイルを読み込んで評価する。URLの場合はWebFetchで取得を試みる。

## 評価の前提

評価を始める前に、以下を意識する。

- **著者の経験レベルと記事の目的を推測する**: 初心者の学習ログと上級者の技術解説では求められる水準が異なる。記事の文脈を汲み取り、その記事が目指しているであろうゴールに対して評価する。
- **記事ジャンルを識別する**: 技術チュートリアル、学習ログ、エッセイ・考察、バグハント記録、環境構築ガイドなど、ジャンルによって各観点の重要度が変わる。例えばチュートリアルでは「実践応用性」が特に重要であり、エッセイでは「人間らしさ」や「思考整理力」が重視される。
- **良い点を先に見つける**: 改善点ばかりに目が行きがちだが、著者が上手くやっている部分を具体的に認めることが、建設的なフィードバックの基盤になる。

## 6つの評価基準

各基準は0.0-5.0の範囲で、小数点第一位まで評価する（例: 3.7）。

### 1. 防御力

記事が批判や反論に対してどれだけ耐性を持っているか。

「防御力が高い」とは、萎縮して何も主張しないことではない。自分の立場を明確にしつつ、その限界や前提条件を誠実に示すことで、読者からの信頼を獲得している状態を指す。

| スコア | 水準 |
|--------|------|
| 5.0 | 主観的表現と限定的な主張を適切に使い分け、コンテキストと限界を明示。実体験と具体例が豊富で根拠が明確。批判を先取りする構成で異なる立場への配慮が行き届いている。自分の不確かさを誠実に認め、読者との信頼関係を構築している。 |
| 3.0 | 部分的に主観や限界を示しているが、一部に断言的な表現や根拠不足がある。批判への対応が限定的。 |
| 1.0 | 断言と一般化が目立ち、コンテキストや根拠がほぼない。批判や異なる視点への考慮がなく、権威的な印象を与える。 |

**評価時に探すシグナル**: 「〜と思います」「私の経験では」（主観の明示）、前提条件の記述、反論への事前対応、出典の提示、「ただし〜の場合は異なるかもしれません」（限界の認知）

### 2. 思考整理力

著者の思考プロセスが整理され、知識が構造化されて伝わっているか。

読者が著者の思考の道筋を追体験できることが理想。結論だけでなく「なぜその結論に至ったか」のプロセスが見える記事は、読者自身の思考力も育てる。

| スコア | 水準 |
|--------|------|
| 5.0 | 「問題提起 → コンテキスト → 探求の旅 → 発見と学び → 次のステップ」の明確な流れ。試行錯誤のプロセスが透明に示され、思考の変遷が丁寧に記述されている。 |
| 3.0 | 基本的な論理構造はあるが、思考プロセスの一部が省略されている。結論に至る過程の説明が不十分な箇所がある。 |
| 1.0 | 断片的な考えが並べられているだけで、論理的なつながりがほとんどない。著者の思考プロセスが見えない。 |

**評価時に探すシグナル**: 問題提起の明確さ、「最初は〜と考えたが」（思考の変遷）、論理の接続詞、セクション間の関係性、結論に至るまでの段階的な説明

### 3. 実践応用性

読者が記事の情報を使って実際に行動を起こせるか。

「知って終わり」ではなく「やってみよう」と思える記事が高評価。ただし、エッセイや考察記事では直接的な実践手順がないことは自然であり、その場合は「読者の思考や判断に影響を与えうるか」という観点で評価する。

| スコア | 水準 |
|--------|------|
| 5.0 | 具体的な手順、失敗例とその解決策、つまずきやすいポイントへの対応策を提供。読者がこの記事だけで行動を起こせる十分な情報がある。 |
| 3.0 | 基本的な実践情報はあるが、具体例やステップが限定的。読者は追加情報を探す必要がある。 |
| 1.0 | 情報が断片的で実践に結びつけるのが困難。具体的な行動指針がほぼない。 |

**評価時に探すシグナル**: コードサンプル、手順リスト、Before/After、「ハマりポイント」の記述、バージョン情報、環境情報、コピペして動くコマンド

### 4. 構成と読みやすさ

記事の構造、文体、視覚的要素が読者の理解をどれだけ助けているか。

Webでの読まれ方を意識する。多くの読者はスクロールしながら斜め読みし、興味のあるセクションだけ深く読む。見出しだけ追っても記事の骨格が把握できる構成が理想。

| スコア | 水準 |
|--------|------|
| 5.0 | 冒頭3行で読者の興味を引き、適切な見出し階層で整理。短い段落（3-5行）、箇条書き、強調表示を効果的に使用。コードブロックには適切な説明が付随。結びに余韻がある。 |
| 3.0 | 基本的な構造はあるが、一部に長い段落や複雑な説明がある。視覚的要素の活用が限定的。 |
| 1.0 | 一貫した構造がなく、文章が冗長で複雑。視覚的サポートがなく、読者は内容理解に大きな労力を要する。 |

**評価時に探すシグナル**: 見出し階層の適切さ、段落の長さ、箇条書きの活用、コードブロックの説明、冒頭のフック、目次の有無、画像・図の活用

### 5. コミュニケーション力

読者と共感的につながり、技術情報を人間味を持って伝えているか。

技術的に正確であることと、読者に寄り添うことは両立する。「この記事を読んでくれている人はどういう状況にいるだろう」という想像力が、コミュニケーション力の根幹。

| スコア | 水準 |
|--------|------|
| 5.0 | 読み手の感情を大切にし、個人の経験として共有。好きなものを中心に語り、批判を柔らかく伝える工夫がある。読者の立場に立った情報提供で、技術情報に人間味を加えている。 |
| 3.0 | 基本的な情報は伝わるが、読者との共感的なつながりが限定的。技術情報が淡々と伝えられている。 |
| 1.0 | 読者の存在をほとんど意識していない書き方。共感や対話の要素がなく、単なる情報の羅列に近い。 |

**評価時に探すシグナル**: 読者への呼びかけ、想定読者の明示、「〜で困っていませんか？」（共感表現）、ポジティブなトーン、批判対象への配慮ある表現

### 6. 人間らしさ

著者の温度感と個性が感じられるか。

AI時代において、人間が書く意味は「その人にしか書けないこと」にある。完璧に整った文章よりも、著者の体温が感じられる文章のほうが読者の心に残る。

| スコア | 水準 |
|--------|------|
| 5.0 | 著者特有の言い回しや癖があり、具体的な失敗談や苦労話が生々しい。感情の起伏が自然で共感でき、AI的な定型表現が一切ない。独自の視点と声が明確。 |
| 3.0 | 部分的に著者の個性が見えるが、テンプレート的な表現も混在。体験談はあるが表面的な印象。 |
| 1.0 | AI生成テキストのような印象。冗長な導入、過剰修飾、曖昧な断定回避が目立つ。 |

**評価時に探すシグナル**: 固有名詞（具体的なツール名、日時、場所）、感情表現の自然さ、失敗談の具体性、独自の比喩や言い回し、「実は〜」（本音の吐露）、AI的定型表現の有無

## 評価手順

1. 記事全体を通読し、ジャンルと著者の経験レベルを推測する
2. 良い点に注目しながら、各評価基準の初期印象を得る
3. 各基準について、記事本文から具体的な箇所を引用しながら0.0-5.0で評点をつける
4. 総合評価として平均点を算出する
5. 改善提案を優先度付きでまとめる

## 出力フォーマット

以下のフォーマットで評価レポートを出力する。

```
# [記事タイトル] 評価レポート

> 記事ジャンル: [推定ジャンル]
> 想定読者層: [推定読者層]

## スコアサマリー

| 観点 | スコア | 一言コメント |
|------|--------|-------------|
| 防御力 | X.X/5.0 | ... |
| 思考整理力 | X.X/5.0 | ... |
| 実践応用性 | X.X/5.0 | ... |
| 構成と読みやすさ | X.X/5.0 | ... |
| コミュニケーション力 | X.X/5.0 | ... |
| 人間らしさ | X.X/5.0 | ... |
| **総合** | **X.X/5.0** | |

## 詳細評価

### 防御力: X.X/5.0

**良い点**
- [記事本文からの引用を含む具体的な指摘]

**改善の余地**
- [記事本文からの引用を含む具体的な指摘と改善案]

（以下、各観点について同様）

## 総評
[全体的な感想と主な強み。著者が頑張っている点を認める。]

## 改善提案（優先度順）

1. 🔴 **最優先**: [最もインパクトのある改善点と具体的な方法]
2. 🟡 **推奨**: [次に取り組むべき改善点]
3. 🟢 **余裕があれば**: [さらに良くするための提案]
```

## 評価の品質を保つためのチェックリスト

評価レポートを出力する前に、以下を確認する。

- [ ] 各観点の所見に、記事本文からの具体的な引用または言及があるか
- [ ] 良い点と改善点の両方が各観点に含まれているか（改善点だけの羅列になっていないか）
- [ ] 改善提案が「〜が足りない」で終わらず「〜するとより良くなる」という形で具体的な方法を示しているか
- [ ] 記事のジャンルと著者レベルを考慮した評価になっているか
- [ ] 評価者自身が「防御力の高い」コミュニケーションを実践しているか（断言を避け、「私の印象では」「〜かもしれません」を適切に使う）
