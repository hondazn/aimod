Fast is fine, but accuracy is everything.

# 開発スタイル

- Issue → 精緻化 → ブランチ → コーディング → PR の順で進める
- TDD でコーディングする（探索 → Red → Green → Refactoring）
- KPI やカバレッジ目標が与えられたら達成まで試行する
- ユーザーの意図と指示の理解を最優先し、不明瞭な前提は質問して明確にする

# コード設計

- 関心の分離を保つ
- 状態とロジックを分離する
- 冪等性、可読性、保守性を重視する
- コントラクト層（API/型）を厳密に定義し、実装層は再生成可能に保つ
- 静的検査可能なルールはプロンプトではなく、その環境の linter か ast-grep で記述する

# 委譲ワークフロー

実装・修正・リファクタ・調査を含む作業は、応答や他ツール呼び出しに先立って必ず **dev-orchestration** スキルを起動し、その判定に従って codex（調査・レビュー）/ cursor（実装）に委譲する。エージェント役割、判定トリガー、協力ループ（plan → build → verify）、例外パス、レッドフラグの詳細はスキル本文を参照。ソースコードへの直接 Edit/Write は `~/.claude/hooks/delegate-coding.sh` で物理ブロックされる（バックストップ）。
