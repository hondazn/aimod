# README 作成スキル

プロジェクトのREADME・概要説明を作成・改善する。

READMEの本質は「このプロジェクトを初めて見た人が、30秒以内に"何ができるか"を理解し、5分以内に動かせる」こと。この原則に沿って、情報の取捨選択を行う。

## モード選択

入力に応じて以下のモードを判断:

| モード | トリガー | 出力 |
|--------|----------|------|
| **作成** | 「書いて」「作って」「生成して」+ プロジェクト情報 | 完成したREADME |
| **レビュー** | 既存READMEが提示 + 「レビュー」「改善」「チェック」 | 改善提案 + 修正版 |
| **テンプレート** | 「テンプレート」「ひな形」「フォーマット」 | 空のテンプレート |

## 作成モード

### Step 1: コードベースの分析

README作成の前に、まずプロジェクトの実態を把握する。ユーザーの説明だけに頼らず、実際のコードから情報を読み取ることで、正確で実用的なREADMEが書ける。

以下を実行:

1. **プロジェクト構造の把握**: ディレクトリ構成、主要ファイルを確認
2. **技術スタックの特定**: `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `Gemfile` 等から言語・フレームワーク・依存関係を読み取る
3. **エントリーポイントの確認**: `main`, `index`, `app` 等のファイルから実際の使い方を把握
4. **既存ドキュメントの確認**: 既にREADMEやdocsがあれば内容を確認
5. **設定ファイルの確認**: CI/CD、Docker、設定ファイルから動作環境の要件を把握

### Step 2: 情報の整理と確認

分析結果をもとに、以下の情報を整理する。不足している情報はユーザーに質問する:

- プロジェクト名
- 何をするものか（一文で。動詞で始める）
- 主要機能（3〜5個。数値や具体的な差別化ポイントを含める）
- インストール方法（実際のパッケージマネージャーに合わせる）
- 基本的な使用例（コピペで動くもの）
- ライセンス

### Step 3: 構成と執筆

下部の [テンプレート集](#テンプレート集) からプロジェクトの種類に応じたテンプレートを選択し、構成する。

執筆時の原則:

- **冒頭は動詞で始める**: 「〜する」で1行目を完結させる。「このプロジェクトは〜」とは書かない
- **コード例は完全に動くものを**: 変数の定義、import文、期待される出力をすべて含める
- **言語とツールはプロジェクトの実態に合わせる**: npm前提にせず、実際のパッケージマネージャーを使う
- **特徴には具体性を持たせる**: 「高速」ではなく「標準ライブラリの3倍高速」のように数値や比較を入れる

### Step 4: 品質チェック

作成後、以下を確認:

- [ ] 冒頭の一文でプロジェクトの目的が伝わるか
- [ ] インストール手順が実際に動作するか（パッケージ名・コマンドが正確か）
- [ ] コード例がコピペで使えるか（未定義変数や省略がないか）
- [ ] 特徴が3〜5個に絞られ、それぞれに具体性があるか
- [ ] 見出しレベルが適切か（H1はプロジェクト名のみ、H2で主要セクション）
- [ ] 冗長な説明がないか

### Step 5: 確認ループ

作成後、ユーザーに確認:

1. **構成確認**: 「この構成でよろしいですか？」
2. **情報の過不足**: 「追加・削除したい情報はありますか？」
3. **最終レビュー**: 品質チェックの結果を提示

## レビューモード

### 評価基準

| 項目 | チェック内容 | 重視する理由 |
|------|-------------|-------------|
| **目的明確性** | 冒頭で何をするか分かるか | 読者の95%はREADMEの最初の3行で判断する |
| **インストール** | 手順が完全で動作するか | 動かせなければ離脱する |
| **使用例** | コピペで実行できるか | 試せなければ価値が伝わらない |
| **構造** | 見出しレベルが適切か | スキャンしやすさに直結する |
| **簡潔性** | 冗長な説明がないか | 長いほど読まれない |
| **技術的正確性** | バージョンやコマンドが正しいか | 動かない手順は信頼を損なう |

### 品質スコア

| スコア | 状態 |
|--------|------|
| 5.0 | 全項目OK、すぐ使える |
| 4.0 | 軽微な改善点1〜2箇所 |
| 3.0 | 複数の改善点あり |
| 2.0 | 必須要素に欠落あり |
| 1.0 | 大幅な書き直しが必要 |

### 出力形式（レビュー）

```
## レビュー結果

### 品質スコア: X.X / 5.0

### 良い点
- [具体的に評価できる点]

### 改善が必要な点
1. [問題] → [該当箇所の引用] → [具体的な改善案]

---

### 修正版

[修正したREADME全文]
```

## テンプレート集

テンプレート内のプレースホルダーはプロジェクトの実際の言語・ツールに合わせて置き換えること。

### 選択ガイド

| テンプレート | 用途 | 判断基準 |
|-------------|------|---------|
| シンプル版 | 内部ツール、小規模プロジェクト | READMEに2分以上かけたくない |
| 標準版 | 一般的なプロジェクト | チームメンバーや将来の自分が読む |
| 詳細版 | OSSプロジェクト、公開ライブラリ | 不特定多数が利用する |
| ライブラリ向け | npm/pip/cargo等で配布するライブラリ | APIドキュメントが重要 |
| CLIツール向け | コマンドラインツール | コマンドとオプションの説明が中心 |

### シンプル版

最小限のREADME。小規模プロジェクトや内部ツール向け。

```markdown
# プロジェクト名

[何をするか、動詞で始める一文]

## インストール

\`\`\`bash
[実際のインストールコマンド]
\`\`\`

## 使い方

\`\`\`[言語]
[変数定義・import文を含む、コピペで動く完全な例]
\`\`\`

## ライセンス

[ライセンス名]
```

## インストール

\`\`\`bash
[実際のインストールコマンド]
\`\`\`

\`\`\`[言語]
[import文、変数定義、実行、期待される出力を含む完全な例]

### ライブラリ向け

API使用例を重視したテンプレート。

```markdown
# ライブラリ名

[何をするか、動詞で始める一文]

## インストール

\`\`\`bash
[実際のインストールコマンド]
\`\`\`

## API

### `functionName(param)`

[何をする関数か]

**パラメータ:**
- `param` ([型]): [説明]

**戻り値:** [型] - [説明]

**例:**
\`\`\`[言語]
[import、変数定義、実行、出力を含む完全な例]
\`\`\`

### `anotherFunction(options)`

[何をする関数か]

**パラメータ:**
- `options.key1` ([型]): [説明]
- `options.key2` ([型], optional): [説明]

**例:**
\`\`\`[言語]
[完全な例]
\`\`\`

## 完全な例

\`\`\`[言語]
[複数のAPIを組み合わせた実用的なシナリオ]
\`\`\`

## ライセンス

[ライセンス名]
```

### CLIツール向け

コマンド例を重視したテンプレート。

```markdown
# CLI名

[何をするか、動詞で始める一文]

\`\`\`bash
[最も代表的な実行例を1つ]
\`\`\`

## インストール

\`\`\`bash
[実際のインストールコマンド]
\`\`\`

## 使い方

\`\`\`bash
CLI名 [command] [options]
\`\`\`

## コマンド

### `init`

[何をするか]

\`\`\`bash
CLI名 init
CLI名 init --template typescript
\`\`\`

| オプション | 説明 | デフォルト |
|-----------|------|-----------|
| `--template, -t` | テンプレートを指定 | - |

### `build`

[何をするか]

\`\`\`bash
CLI名 build
CLI名 build --watch
\`\`\`

| オプション | 説明 | デフォルト |
|-----------|------|-----------|
| `--watch, -w` | ファイル変更を監視 | false |
| `--output, -o` | 出力先を指定 | ./dist |

## グローバルオプション

| オプション | 説明 |
|-----------|------|
| `--help, -h` | ヘルプを表示 |
| `--version, -v` | バージョンを表示 |
| `--verbose` | 詳細ログを出力 |

## 設定ファイル

`cli名.config.json` で設定をカスタマイズ:

\`\`\`json
{
  "output": "./dist",
  "port": 8080
}
\`\`\`

## ライセンス

[ライセンス名]
```

## 避けるべきパターン

| 避ける | 代わりに | 理由 |
|--------|----------|------|
| 「このプロジェクトは〜について...」 | 「[動詞]する」で始める | 読者は目的を即座に知りたい |
| 長いFeatureリスト（10個以上） | 主要な3〜5個に絞る | 多すぎると印象に残らない |
| 動かないコード例 | 変数定義と出力を含む完全な例 | 試せないコードは信頼を損なう |
| バージョン依存の説明 | 別ドキュメントにリンク | メンテナンスコストが高い |
| スクリーンショットだらけ | 必要最小限に | ロードが遅く、検索できない |
| npm前提のインストール手順 | プロジェクトの実態に合わせる | 読者を混乱させる |

具体的な例は [EXAMPLES.md](EXAMPLES.md) を参照。
