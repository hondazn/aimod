# チュートリアル作成スキル

ステップバイステップのチュートリアル・ガイドを作成・改善する。

## モード選択

入力に応じて以下のモードを自動判断する。

| モード | トリガー | 出力 |
|--------|----------|------|
| **作成** | 「書いて」「作って」「教えて」+ 達成目標 | 完成したチュートリアル |
| **レビュー** | 既存チュートリアルが提示 + 「レビュー」「改善」「添削」 | 評価 + 改善提案 + 修正版 |
| **テンプレート** | 「テンプレート」「ひな形」「フォーマット」 | 空のテンプレート（[TEMPLATES.md](TEMPLATES.md) から選択） |

---

## 作成モード

### Step 1: 情報収集

以下を確認する。不足している場合は質問して埋める（一度に全部聞かず、最も重要な2-3項目に絞る）。

| 項目 | 質問例 | 理由 |
|------|--------|------|
| **達成目標** | 「読者は最終的に何ができるようになりますか？」 | ゴールが曖昧だとステップ分解ができない |
| **対象読者** | 「想定読者の技術レベルは？」 | 説明の粒度と用語選択に直結する |
| **前提条件** | 「読者に求める事前知識や環境は？」 | 読者が「準備できない」チュートリアルは完走できない |
| **所要時間** | 「目安の所要時間は？」 | ステップ数と深さの設計指針になる |
| **成果物の形式** | 「Markdownファイル？社内Wiki？Zenn記事？」 | フォーマットやメタデータの要否が変わる |

ユーザーが「とにかく早く作って」という雰囲気の場合は、最小限の確認（達成目標と対象読者）だけで進める。

### Step 2: 構造設計

目標達成までのステップを分解する。設計原則は3つ。

**1ステップ1アクション**: 複数のことを1ステップに詰め込むと、途中で失敗したとき原因特定が困難になる。読者が「今どこにいるか」を常に把握できる粒度にする。

**各ステップに確認ポイント**: 読者は「自分が正しくできたか」を知りたい。「期待される出力」「確認コマンド」「画面に表示されるべきもの」を必ず示す。確認方法のないステップは、読者にとって暗闇を歩くようなものになる。

**依存関係の明示**: あるステップが前のステップの成果物に依存する場合、その関係を明記する。読者が途中からやり直す場合の助けになる。

### Step 3: 執筆

| 原則 | 理由 |
|------|------|
| 命令形で書く（「〜する」） | 「〜してください」より簡潔で、手順書として読みやすい |
| コードは完全に書く | 省略（`...`）があるとコピペで動かない。読者の最大のフラストレーション源 |
| 期待される出力を示す | 読者が「合っているか」を判断できる唯一の手段 |
| よくあるエラーと対処を書く | 読者の80%が詰まるポイントは予測可能。先回りすることで完走率が劇的に上がる |
| 冒頭で「何が作れるか」を見せる | 読者のモチベーション維持。完成形が見えると途中で離脱しにくい |

### Step 4: セルフレビュー

作成後、以下のチェックリストで品質を確認する。特に以下の3点は致命的な問題になりやすい。

1. **コードの実行可能性**: すべてのコード例がコピペで動くか（import文、変数定義、依存パッケージの漏れ）
2. **ステップの完全性**: 前提知識がない読者が、Step 1から順に実行して最後まで到達できるか
3. **確認ポイントの存在**: すべてのステップに「成功したことを確認する方法」があるか

上記3点を含む全チェック項目を以下に示す。「必須」が全てOKになるまでは公開しない。

#### 必須項目（これが欠けると読者が完走できない）

- [ ] 達成目標が冒頭で明記されている（「このチュートリアルでは〜ができるようになります」）
- [ ] 前提条件（環境・知識・バージョン）が具体的に書かれている
- [ ] 所要時間の目安がある
- [ ] すべてのコードがコピペで動作する（import文、変数定義、依存パッケージの漏れなし）
- [ ] コード内に省略（`...`）がない
- [ ] 各ステップに確認方法がある（期待される出力、確認コマンド、画面表示のいずれか）

#### 構造（読者が迷わず進めるか）

- [ ] 1ステップ1アクションになっている
- [ ] ステップが正しい順序で並んでいる（依存関係が逆転していない）
- [ ] 前のステップの結果を使う場合、その依存関係が明記されている
- [ ] トラブルシューティングセクションがある（最低でもよくある1-2個のエラー）

#### コード品質（動くコードか）

- [ ] 使用言語/ランタイムのバージョンが明記されている
- [ ] 変数が全て定義されている（未定義の変数への参照がない）
- [ ] import文/require文が含まれている
- [ ] 期待される出力が示されている
- [ ] エラーハンドリングがある（本番コードに近い場合）

#### 読者体験（理解と定着を助けるか）

- [ ] 専門用語に説明がある（初心者向けの場合）
- [ ] 「なぜ」この操作をするか説明がある（初心者向けの場合）
- [ ] 完成イメージが冒頭にある（読者のモチベーション維持）
- [ ] まとめ・学んだことがある（振り返りで理解が定着する）
- [ ] 次のステップへの導線がある（学習の継続を促す）

#### アンチパターン（該当しないことを確認）

- [ ] 「まず〜について理解しましょう」で始まっていない
- [ ] 1ステップに複数アクションが詰め込まれていない
- [ ] 「うまくいくはずです」のような曖昧な確認表現がない
- [ ] `...` で省略されたコードブロックがない
- [ ] 抽象的な説明だけのステップがない（コードや具体例が伴っている）

#### クイック検証（5分でできるチェック）

時間がないときは以下の4点だけ確認する。

1. **冒頭を読む** - 何ができるようになるか、一文で分かるか？
2. **ステップ数を数える** - 3〜10ステップに収まっているか？（超える場合は分割を検討）
3. **コードブロックを眺める** - 各ブロックが単独でコピペ可能か？ `...` がないか？
4. **確認ポイントを探す** - 各ステップに「期待される出力」や「確認」の記述があるか？

#### チュートリアル種別ごとの追加チェック

**入門チュートリアル**
- [ ] 専門用語すべてに説明がある
- [ ] UI操作の場合、スクリーンショットまたは具体的な画面説明がある
- [ ] 「なぜ」の説明が各ステップにある
- [ ] 環境構築手順が省略されていない

**クイックスタート**
- [ ] 5ステップ以内に収まっている
- [ ] 説明は最小限で、コードとコマンドが中心
- [ ] 最短経路で動作する結果が得られる

**ワークショップ**
- [ ] 演習問題がある
- [ ] 解答例がある（折りたたみ推奨）
- [ ] 時間配分が明記されている
- [ ] セクションごとに独立して実施できる構成

**トラブルシューティングガイド**
- [ ] 症状/エラーメッセージで逆引きできる構成
- [ ] 各問題に原因・解決方法・予防策がある
- [ ] デバッグの一般的な手順が示されている

### 出力形式

```
# チュートリアル: [達成目標]

[何ができるようになるか、一文で]

## 完成イメージ
[最終的な成果物のスクリーンショットや出力例]

## 前提条件
- [必要な環境・知識]

## 所要時間
約XX分

---

## ステップ

### 1. [アクション]
[説明]
[コード]
**確認**: [成功条件]

### 2. [アクション]
...

---

## トラブルシューティング
### [エラーメッセージ/症状]
**原因**: ...
**解決方法**: ...

---

## まとめ
このチュートリアルで学んだこと:
- [ポイント1]
- [ポイント2]

## 次のステップ
- [発展的な内容へのリンク]
```

作成後、末尾に以下のメタ情報を添える。

```
---
### 作成メモ
- 対象読者: [初心者/中級者/上級者]
- 所要時間: [目安]
- ステップ数: [N個]
- 主な技術: [使用する技術/ツール]
```

## レビューモード

### 評価の進め方

既存のチュートリアルが提示されたら、以下の観点で評価する。

| 観点 | チェック内容 | 重要度 |
|------|-------------|--------|
| **目標の明確性** | 冒頭で「何ができるようになるか」が一文で分かるか | 高 |
| **前提条件** | 必要な環境・知識が具体的に示されているか | 高 |
| **ステップの粒度** | 1ステップ1アクションになっているか | 高 |
| **コードの完全性** | コピペで動くか、省略がないか | 高 |
| **確認ポイント** | 各ステップで「成功したか」を読者が判断できるか | 高 |
| **トラブル対応** | よくあるエラーへの対処があるか | 中 |
| **まとめ/次のステップ** | 学んだことの振り返りと発展への導線があるか | 低 |

### 品質スコア

| スコア | 状態 | 目安 |
|--------|------|------|
| 5.0 | 初見の読者でも一人で完走できる | 全チェック項目OK |
| 4.0 | 軽微な補足で完走可能 | 高重要度は全てOK、中以下に1-2個の問題 |
| 3.0 | 一部ステップが不明確 | 高重要度に1-2個の問題 |
| 2.0 | 動作しないコード/手順飛び | 致命的な問題がある |
| 1.0 | 目標達成不可能 | 大幅な書き直しが必要 |

### レビュー出力形式

```markdown
## レビュー結果

### 品質スコア: X.X / 5.0

### 良い点
- [具体的に評価できる点]

### 改善が必要な点
1. **[問題の要約]** (該当: Step N)
   - 現状: [何が問題か]
   - 提案: [どう直すか]
   - 理由: [なぜ直すべきか]

---

### 修正版
[改善を反映したチュートリアル全文]
```

## テンプレートモード

[TEMPLATES.md](TEMPLATES.md) から用途に合ったテンプレートを選んで提供する。

| テンプレート | 用途 | 対象読者 |
|-------------|------|----------|
| 標準 | 新機能の導入、初めての利用 | 初心者〜中級者 |
| クイックスタート | 概要把握、素早く試したい | 中級者〜上級者 |
| ワークショップ | 実践的な学習、研修、勉強会 | 全レベル |
| トラブルシューティング | 問題解決、サポート文書 | 全レベル |

ユーザーが用途を明示しない場合は、「どのような場面で使いますか？」と確認してから選択する。

## 読者レベル別の調整

読者レベルによって説明の粒度と内容を変える。これはチュートリアルの品質に最も影響する要素の一つ。

### 初心者向け
- 専門用語には必ず簡潔な説明を添える（括弧書きまたは脚注）
- ステップを細かく分割する（迷う余地を減らす）
- 「なぜ」この操作をするか説明する（理解が定着につながる）
- 環境構築の手順も省略しない
- エラー時の対処を手厚くする（初心者は小さなエラーで離脱しやすい）

### 中級者向け
- 概念説明は最小限（知っている前提）
- 実践的な例を重視する
- ベストプラクティスや設計上の判断理由を示す
- 応用のヒントを添える

### 上級者向け
- 本質のみ簡潔に書く
- 応用例やカスタマイズ方法を充実させる
- パフォーマンス考慮事項や内部動作の解説を含める
- リファレンス的に使えるよう構成する

## 避けるべきパターン

| 避ける | 代わりに | 理由 |
|--------|----------|------|
| 「まず〜について理解しましょう」で始める | すぐ手を動かす内容から始める | 読者は「動くもの」を見たくて来ている |
| 1ステップに複数アクション | 1ステップ1アクション | エラー時の原因特定が困難になる |
| 「うまくいくはずです」 | 具体的な確認コマンドや出力例 | 読者は「はず」では安心できない |
| コードの省略 `...` | 完全なコード | コピペで動かないコードは信頼を失う |
| エラー時の対処なし | トラブルシューティングセクション | 読者の80%は同じところで詰まる |
| 抽象的な説明だけ | 具体的なコード例 | 「分かった気になる」だけで手が動かない |

具体的な良い例・悪い例は [EXAMPLES.md](EXAMPLES.md) を参照。
