## ワークフロー概要

入力に応じてモードを判断し、対応するワークフローを実行する。

| モード | トリガー | 出力 |
|--------|----------|------|
| **作成** | コードまたはAPI情報 + 「書いて」「作って」「ドキュメント化して」 | 完成したリファレンス |
| **レビュー** | 既存ドキュメント + 「レビュー」「改善」「チェックして」 | 品質スコア + 改善提案 + 修正版 |
| **テンプレート** | 「テンプレート」「ひな形」「スケルトン」+ 種類指定 | 記入用テンプレート |

判断に迷う場合はユーザーに確認する。複数の対象が含まれている場合は、それぞれ個別にリファレンスを作成する。

## 作成モード

### Step 1: 対象の特定と情報抽出

対象の種類を判断し、コードから必要な情報を抽出する。情報が不足している場合は推測せずユーザーに質問する。これは正確なドキュメントの基盤となるため、曖昧なまま進めない。

**関数/メソッド**:
- 関数名、所属（モジュール/クラス）
- パラメータ（名前、型、必須/任意、デフォルト値、制約）
- 戻り値（型、説明、nullableかどうか）
- 例外/エラー（型、発生条件）
- 副作用（ある場合）
- スレッドセーフ性（並行処理が関係する場合）

**クラス/インターフェース**:
- クラス名、継承関係、実装インターフェース
- コンストラクタ（パラメータ、制約）
- パブリックメソッド（上記の関数と同じ項目）
- パブリックプロパティ（型、読み取り専用かどうか、説明）
- イベント（発行する場合）

**REST API**:
- HTTPメソッド + パス（パスパラメータを含む）
- 認証方式（Bearer Token、API Key、OAuth等）
- リクエストヘッダー、クエリパラメータ、ボディ
- レスポンス（成功/エラー、ステータスコード別）
- レート制限（ある場合）
- ページネーション方式（ある場合）

**GraphQL**:
- Query/Mutation/Subscription名
- 引数（型、必須/任意）
- 戻り値の型（フィールド一覧）
- ディレクティブ

### Step 2: 使用例の作成

使用例はドキュメントの中で最も読まれる部分。コピー&ペーストしてそのまま動くコードを書く。

- **基本的な使用例**（必須）: 最も一般的なユースケースを1つ示す
- **オプション付きの例**: オプショナルパラメータの使い方を示す
- **エラーハンドリングの例**: 主要なエラーのキャッチと処理を示す
- **実際の出力**: 各例に期待される出力をコメントで付記する

### Step 3: ドキュメント構成

[STYLE-GUIDE.md](STYLE-GUIDE.md) に従って一貫したスタイルで記述する。構成の基本:

1. 関数名/エンドポイント（見出し）
2. 一文の要約（動詞で始める）
   - 動詞で始め、「する」で終わる。主語（「この関数は」等）を省略して簡潔にする
   - 良い例: `ユーザーを取得する。` / `新しい注文を作成する。`
   - 避ける: `ユーザーを取得します。`（「〜する」に統一）/ `この関数はユーザーを取得する。`（主語は不要）/ `ユーザーの取得`（体言止めは避ける）
3. シグネチャ/エンドポイント定義
4. パラメータ表
5. 戻り値/レスポンス
6. 使用例
7. エラー一覧
8. 関連API

### Step 4: 確認

作成後、以下を確認してユーザーに提示する:

1. **情報の正確性**: パラメータ・戻り値の説明がコードと一致しているか
2. **使用例の動作性**: 例がそのまま実行可能か
3. **網羅性**: 主要なエラーケースやエッジケースが記載されているか

#### 一貫性チェックリスト

ドキュメント全体の一貫性を確認する際に使用する。

- 命名:
  - [ ] パラメータ名がcamelCaseか（Python以外）
  - [ ] 型名がPascalCaseか
  - [ ] エンドポイントがkebab-caseで複数形か
- 説明:
  - [ ] 説明が動詞で始まっているか（「〜する」で終わる）
  - [ ] 同じ概念に同じ用語を使っているか
  - [ ] 冗長な主語（「この関数は」等）がないか
- 表:
  - [ ] すべてのパラメータに型があるか
  - [ ] 必須/任意がYes/Noで統一されているか
  - [ ] デフォルト値が説明欄に記載されているか
- 例:
  - [ ] コード例に未定義変数がないか
  - [ ] コード例に期待される出力があるか
  - [ ] import文が含まれているか

## レビューモード

### 評価基準

| 項目 | チェック内容 | 重み |
|------|-------------|------|
| **完全性** | 必須要素（パラメータ、戻り値、例）が揃っているか | 高 |
| **正確性** | 型情報、デフォルト値、制約がコードと一致するか | 高 |
| **例の品質** | コピペで動作するか、出力が示されているか | 高 |
| **一貫性** | 命名規則・形式がドキュメント全体で統一されているか | 中 |
| **可読性** | 表・見出し・リンクで情報が素早く見つけられるか | 中 |
| **保守性** | コード変更時にドキュメントを更新しやすい構造か | 低 |

### 品質スコア

| スコア | 状態 | 基準 |
|--------|------|------|
| 5.0 | 模範的 | 全項目網羅、例が充実、エッジケースも記載 |
| 4.0 | 良好 | 軽微な改善点が1-2箇所 |
| 3.0 | 改善必要 | パラメータ説明不足、例が不完全 |
| 2.0 | 要修正 | 型情報欠落、動かない例がある |
| 1.0 | 不十分 | 必須要素（パラメータ、戻り値、例のいずれか）が欠落 |

### 出力形式（レビュー）

```
## レビュー結果

### 品質スコア: X.X / 5.0

### 良い点
- [具体的に評価できる点]

### 改善が必要な点
1. **[問題の種類]** - [該当箇所の引用] - [具体的な改善案]

---

### 修正版

[改善を反映した完全なドキュメント]
```

## テンプレートモード

用途に応じたテンプレートを提供する。[TEMPLATES.md](TEMPLATES.md) に各テンプレートの全文がある。

| テンプレート | 用途 | いつ参照するか |
|-------------|------|---------------|
| 関数テンプレート | 単独の関数/メソッド | ユーティリティ関数やヘルパーメソッドの文書化 |
| クラステンプレート | クラス全体 | OOPクラスやサービスクラスの文書化 |
| REST APIテンプレート | HTTPエンドポイント | Web API仕様の作成 |
| OpenAPIテンプレート | 機械可読なAPI仕様 | Swagger UI連携やコード自動生成が必要な場合 |

テンプレートを出力する際は、プレースホルダー（`[説明]` 等）をそのまま残し、ユーザーが記入しやすい形で提供する。

## 言語別の抽出ガイド

コードからAPIドキュメントに必要な情報を抽出する際の言語固有のポイント。

### TypeScript/JavaScript

```typescript
// JSDoc + 型定義から抽出
/**
 * ユーザーを取得する
 * @param id - ユーザーID
 * @returns ユーザー情報
 * @throws UserNotFoundError - ユーザーが存在しない場合
 */
function getUser(id: string): Promise<User> { ... }
```

抽出ポイント:
- JSDocコメント(`@param`, `@returns`, `@throws`, `@deprecated`, `@since`)
- TypeScript型アノテーション（型エイリアス、ジェネリクス、ユニオン型も追う）
- `export` の有無でパブリックAPIかどうかを判断
- `interface` / `type` 定義から戻り値やパラメータの構造を展開

### Python

```python
def get_user(user_id: str) -> User:
    """
    ユーザーを取得する。

    Args:
        user_id: ユーザーID

    Returns:
        ユーザー情報

    Raises:
        UserNotFoundError: ユーザーが存在しない場合
    """
```

抽出ポイント:
- docstring（Google style / NumPy style / reST style を判別）
- 型ヒント（`typing` モジュールの `Optional`, `Union`, `List` 等）
- デコレータ（`@staticmethod`, `@classmethod`, `@property`, `@abstractmethod`）
- `__init__` からコンストラクタパラメータを抽出

### Go

```go
// GetUser retrieves a user by ID.
// Returns ErrUserNotFound if the user does not exist.
func GetUser(id string) (*User, error) { ... }
```

抽出ポイント:
- GoDocコメント（関数名で始まるコメント）
- エクスポート（大文字始まり）の判定
- 多値返却（特に `error` 型）のドキュメント化
- `context.Context` パラメータの扱い（通常は第1引数、説明は簡潔に）

### Rust

```rust
/// ユーザーを取得する。
///
/// # Arguments
/// * `id` - ユーザーID
///
/// # Returns
/// ユーザー情報、存在しない場合は `None`
///
/// # Errors
/// データベース接続に失敗した場合
pub fn get_user(id: &str) -> Result<Option<User>, DbError> { ... }
```

抽出ポイント:
- `///` ドキュメントコメント（`# Arguments`, `# Returns`, `# Errors`, `# Panics`, `# Examples`）
- `pub` の有無でパブリックAPIを判定
- `Result<T, E>` / `Option<T>` のエラーセマンティクスを明示
- ライフタイムパラメータの説明（複雑な場合）

## 避けるべきパターン

| 避ける | 代わりに | 理由 |
|--------|----------|------|
| 「この関数は〜します」 | 「〜する」で始める | 冗長さを避け、スキャンしやすくするため |
| 型なしのパラメータ説明 | 型を明示 | 利用者が正しい値を渡すために必要 |
| `...` で省略したコード例 | 完全に動作するコード例 | コピペで動かないと信頼を失うため |
| 戻り値の型だけ | 型 + 何を表すかの説明 | 型だけでは意味が伝わらない場合が多い |
| エラーケースの省略 | 主要なエラーを条件とともに列挙 | エラーハンドリングは利用者が最も必要とする情報 |
| 内部実装の詳細な説明 | 利用者が必要とする振る舞いの説明 | リファレンスは「何ができるか」を伝えるもの |

具体的な良い例・悪い例の対比は [EXAMPLES.md](EXAMPLES.md) を参照。
