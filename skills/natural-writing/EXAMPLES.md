## 1. AI臭い導入の除去

### Before
```
本記事では、プログラミング学習の効果的な方法について
詳しく説明していきたいと思います。
プログラミングは現代社会において非常に重要なスキルであり、
多くの人々がその習得を目指しています。
```

### After
```
プログラミングを学び始めて3ヶ月。
毎日30分のコード練習が、仕事の自動化に直結した。
```

### 判断根拠
- 「本記事では〜説明していきたいと思います」→ 読者はすでにこの記事を開いている。何を書くかの宣言は不要
- 「非常に重要なスキル」「多くの人々」→ 情報量ゼロ。誰が読んでも書ける文
- 著者の実体験から始めることで「この人の話を聞こう」と思える冒頭になる

## 2. 過剰修飾の削減

### Before
```
このツールは非常に優れた機能を持っており、
極めて効率的な作業を実現することができます。
まさに画期的なソリューションと言えるでしょう。
```

### After
```
このツールで作業時間が半分になった。
特にファイル整理の自動化が効いている。
```

### 判断根拠
- 「非常に」「極めて」「まさに」→ 3つの強調語が3文に1つずつ。根拠なく盛っている印象
- 「画期的なソリューション」→ 抽象的な評価。読者は「具体的に何が？」と思う
- 「作業時間が半分」「ファイル整理の自動化」→ 読者が効果を実感できる

## 3. 曖昧表現の具体化

### Before
```
この方法は効果があるかもしれません。
一般的には良い結果が得られると思われます。
基本的にはおすすめできると言えるでしょう。
```

### After
```
この方法で3人中2人が目標を達成した。
ただし、経験者向け。初心者には別のアプローチを勧める。
```

### 判断根拠
- 「〜かもしれません」「〜と思われます」「〜と言えるでしょう」→ 3文連続の断定回避。読者は「結局おすすめなの？」と感じる
- 「一般的には」「基本的には」→ ヘッジ表現の重畳。条件を明示すれば不要
- 対象者を「経験者向け」「初心者には別」と明示することで、曖昧さなく推薦できる

## 4. テンプレート表現の個性化

### Before
```
コミュニケーションが大切です。
チームワークを意識することが重要です。
日々の努力が必要です。
```

### After
```
週1回の15分ミーティングでチームの空気が変わった。
「困ってること何かある？」の一言で、問題が早期に見つかる。
```

### 判断根拠
- 「大切」「重要」「必要」→ 同じ構文パターンが3回。フォーマットで文章を埋めている状態
- 抽象的な心構えは読者の行動を変えない。具体的な行動（週1回15分、一言の問いかけ）とその効果を書くことで、読者が「試してみよう」と思える
- 著者の体験として書くことで、テンプレート感が消える

## 5. 感情押し付けの事実化

### Before
```
この素晴らしい技術は、驚くべき進歩を遂げています。
感動的な成果が次々と生まれており、
誰もが興奮せずにはいられません。
```

### After
```
この技術は5年で処理速度が100倍になった。
医療診断の精度が人間の専門医を超えた分野もある。
```

### 判断根拠
- 「素晴らしい」「驚くべき」「感動的」「興奮せずにはいられません」→ 4つの感情表現。読者の感情を押し付けている
- 感情は事実を読んで読者が抱くもの。「100倍」「専門医を超えた」という事実のほうが、「驚くべき」という形容詞より読者の感情を動かす
- 読者の感情を代弁する（「誰もが興奮」）のは特に避ける。読者は自分の感情を決められたくない

## 6. 文体統一

### Before（混在）
```
まず、設定画面を開きます。
次に、オプションを選択する。
最後に、保存ボタンを押してください。
```

### After（統一: です/ます調）
```
まず、設定画面を開きます。
次に、オプションを選択します。
最後に、保存ボタンを押します。
```

### After（統一: だ/である調）
```
まず、設定画面を開く。
次に、オプションを選択する。
最後に、保存ボタンを押す。
```

### 判断根拠
- 3文中、1文目「〜ます」、2文目「〜する」、3文目「〜ください」と3つの文体が混在
- 短い手順文でも文体の揺れは目につく。読者は無意識に「雑に書かれた文章だ」と感じる
- どちらの文体に統一するかは文書の目的と読者による

## 7. 長文の分割

### Before
```
このプロジェクトは、当初の予定よりも大幅に遅れており、
その原因としては、要件の変更が頻繁に発生したことと、
チーム内のコミュニケーション不足があったと考えられ、
今後は週次のレビューを導入することで改善を図りたいと思います。
```

### After
```
プロジェクトは予定より2ヶ月遅れた。

原因は2つ。要件変更が月平均5回発生したこと。
チーム間の情報共有が週1回しかなかったこと。

対策として、週次レビューを導入する。
```

### 判断根拠
- 原文は1文で4つのことを言っている（遅延の事実、原因1、原因2、対策）。読者は途中で迷子になる
- 「一文一意」の原則を適用し、情報を構造化
- 「大幅に遅れ」→「2ヶ月遅れ」、「頻繁に」→「月平均5回」に具体化
- 「と考えられ」「図りたいと思います」→ ヘッジ表現を削除し、断言

## 8. 専門用語の配慮

### Before
```
APIエンドポイントをRESTfulに設計し、
JWTによる認証を実装してCORSを設定した。
```

### After（一般読者向け）
```
外部サービスとの接続口（API）を設計した。
ログイン認証の仕組みと、異なるサイト間での
データやり取りの許可設定も実装した。
```

### After（技術者向け）
```
RESTful APIを設計。認証はJWT、CORS設定も完了。
```

### 判断根拠
- 同じ事実でも、読者によって最適な表現が異なる
- 一般読者向け: 専門用語を噛み砕き、括弧で原語を添える。何をしているかがわかることを優先
- 技術者向け: 専門用語はそのまま使い、冗長な説明は省く。簡潔さを優先
- 読者が誰かわからない場合は、一般読者向けに書くのが安全

## 9. 複合的な校正

実際の校正では複数の問題が同時に存在する。ここでは複合的なケースを示す。

### Before
```
本日は、弊社の新しいプロジェクト管理ツールについて
ご紹介させていただきたいと思います。
このツールは非常に画期的なもので、
チームの生産性を大幅に向上させることが可能です。
基本的には、どなたでも簡単に使えるものとなっておりますので、
ぜひ一度お試しいただければ幸いです。
```

### After
```
新しいプロジェクト管理ツールを紹介します。

特徴は3つ。タスクの進捗が一目でわかるダッシュボード、
ドラッグ操作だけで担当者を変更できるUI、
Slackへの自動通知。導入チームでは週次の進捗確認が
30分から5分に短縮されました。

無料トライアルが30日間あります。
```

### 判断根拠
- **AI臭い導入の除去**: 「ご紹介させていただきたいと思います」→「紹介します」に簡潔化
- **過剰修飾の削減**: 「非常に画期的」→ 削除し、代わりに具体的な3つの特徴を列挙
- **冗長表現の刈り込み**: 「向上させることが可能です」「ものとなっております」→ 簡潔な表現に
- **具体化**: 「大幅に向上」→「30分から5分に短縮」
- **テンプレート的な締めの排除**: 「お試しいただければ幸いです」→ 具体的な情報（30日間無料トライアル）に

## 校正の思考プロセス（実演）

以下は、文章を校正する際の実際の思考を再現したもの。

### 入力文

```
今回は、効果的な会議の進め方について考えてみたいと思います。
会議は多くのビジネスパーソンにとって非常に重要なものですが、
一般的には無駄が多いと感じている人も少なくないのではないでしょうか。
効率的な会議を実現するためには、いくつかのポイントを
押さえることが必要です。
```

### 思考の流れ

```
1. 全体を読む
   → 「効果的な会議の進め方」がテーマ。5文で前置きだけ。本題がまだ始まっていない。

2. AI臭さを検出
   → 「今回は〜考えてみたいと思います」: 冗長な導入
   → 「非常に重要なもの」: 過剰修飾
   → 「一般的には」「少なくないのではないでしょうか」: 曖昧な断定回避 x2
   → 「いくつかのポイントを押さえることが必要です」: テンプレート構文

3. 文体チェック
   → です/ます調で統一されている。問題なし。

4. 具体化の余地
   → 「多くのビジネスパーソン」→ 誰？
   → 「無駄が多い」→ 何が無駄？
   → 「いくつかのポイント」→ 何個？何？
   → 全体的に抽象的。具体的な情報がゼロ。

5. 判断
   → この5文は前置きとして全削除が妥当。
   → 代わりに、会議の具体的な問題から始める。
```

### 校正結果

```
1時間の会議で、実際に議論しているのは20分程度。
残りは共有事項の読み上げと、結論の出ない雑談に消える。

3つのルールでこれが変わる。
```

### 変更の要約

| 原文の問題 | 対処 | 根拠 |
|-----------|------|------|
| 5文すべてが前置き | 全削除し、問題提起から開始 | 読者は前置きより具体的な問題に引かれる |
| 「非常に重要」 | 削除 | 会議が重要だと言う必要はない（読者は知っている） |
| 「一般的には」「少なくないのではないでしょうか」 | 具体的な数値に置換 | 「20分程度」のほうが読者の実感に訴える |
| 「いくつかのポイント」 | 「3つのルール」に具体化 | 数を示すことで読者の期待値が定まる |
