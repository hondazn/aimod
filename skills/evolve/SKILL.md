# 自己進化ループ・プロトコル

## 設計思想

このプロトコルの目的は、CLAUDE.mdを「生きたドキュメント」として継続的に改善し、ユーザーにとって最高の協力者であり続けることにある。

ルールは増やすだけでなく減らすことも同じくらい重要。冗長なルールは認知負荷を増やし、本質を覆い隠す。常に「このルールは本当に必要か？既存ルールに包含されないか？」と自問する。

## プロトコル

以下のステップを順番に実行する。

### ステップ1: コンテキスト収集

改訂案を出す前に、現状を正確に把握する必要がある。以下を読み込む。

1. グローバルルール `~/.config/claude/CLAUDE.md` を読む
2. 現在のプロジェクトに `CLAUDE.md` があれば読む
3. `~/.claude/learning/` ディレクトリが存在すれば、最近のログを確認する（存在しなければスキップ）

### ステップ2: 自己省察

直近の対話を以下の4つの観点で振り返る。表面的な評価ではなく、ルールとの因果関係を特定することが重要。

- **成功要因**: ユーザーの期待を上回れた場面はどこか。現行ルールのどの部分が貢献したか
- **失敗要因**: 意図を汲み取れなかった、または回答品質が低かった場面はどこか
- **ルールとの乖離**: ルール不足・ルール違反が原因で発生した問題はあるか
- **改善機会**: ルールが異なっていれば、より良い結果を出せた場面はあるか

省察結果を簡潔にまとめて、ユーザーに共有する（提案の根拠として）。

### ステップ3: 振り返りの記録

省察結果を永続的に記録する。記録があることで、時間をまたいだパターン発見が可能になる。

1. `~/.claude/learning/` ディレクトリが存在しなければ作成する
2. 以下のフォーマットで記録ファイルを作成する
   - ファイル名: `YYYYMMDD-HHMMSS-簡潔なタイトル.md`

```markdown
# 振り返り: [タイトル]

## 何がうまくいったか
- (具体的に)

## 何が改善できるか
- (具体的に)

## 新しく学んだこと
- (具体的に)

## ルール化すべきパターン
- (あれば。なければ「なし」)
```

### ステップ4: ルール改訂案の生成

省察に基づき、CLAUDE.mdの具体的な改訂案を生成する。

改訂は以下の5種類。複数を組み合わせてよい。
- **新規追加**: 本当に新しいルールが必要な場合のみ。既存ルールの拡張で対応できないか先に検討する
- **削除**: 使われていない、または他のルールに包含されるルールを積極的に削除する
- **統合**: 類似の概念を扱う複数のルールを1つにまとめる
- **簡潔化**: 冗長な表現を削り、本質に圧縮する
- **再配置**: 重要度に応じて順序を変える

以下のフォーマットで提示する。

````
## ルール改訂提案

**対象ファイル**: (グローバル / プロジェクト / 両方)

### 改訂の目的
(今回の対話で明らかになった課題と、それをどう解決するか)

### 変更内容
```diff
- (削除する古いルール)
+ (追加・変更するルール)
```

### 影響の見積もり
- ルール総数の変化: (例: 15個 -> 13個)
- 変更理由: (なぜこの変更で振る舞いが改善されるか)
- 他ルールとの整合性: (確認済み / 要調整)
````

複数の独立した変更がある場合は、変更ごとに分けて提示する。ユーザーが個別に承認・却下できるようにするため。

### ステップ5: 承認と適用

1. 改訂案をユーザーに提示し、承認を求める
2. ユーザーが承認した場合（「ok」「適用して」「良い」など）:
   - Editツールで対象ファイルを即座に更新する
   - Readツールで更新結果を確認する
   - 変更が正しく適用されたことをユーザーに報告する
3. ユーザーが修正を求めた場合:
   - フィードバックを反映してステップ4に戻る
4. ユーザーが却下した場合:
   - 理由を記録し、次回の省察に活かす

ファイルの物理的な更新を必ず行うこと。論理的な適用（「以後このルールに従います」）だけでは不十分で、次回のセッションに引き継がれない。

### ステップ6: 完了確認

- 更新後のCLAUDE.md全体を俯瞰し、一貫性が保たれているか確認する
- プロジェクトCLAUDE.mdへの波及が必要な場合は、追加提案するか確認する
- 「他に気になる点はありますか？」とユーザーに確認し、なければプロトコルを終了する
